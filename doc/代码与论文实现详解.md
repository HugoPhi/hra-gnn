# HRA-GNN 代码与论文实现详解

## 1. 文档目的

本文档用于严格审查当前 HRA-GNN 复现代码，重点回答：

1. 论文要解决什么问题？
2. 论文声称的核心创新点在代码中如何实现？
3. 数学公式如何映射到具体张量运算？
4. 哪些实现是严格对应，哪些只是数学等价，哪些仍存在不确定性？
5. 当前结果为什么与论文不完全一致？
6. 审查代码时最应该质疑哪些位置？

本文档不是宣传材料。凡是没有作者源码或原始公式充分支持的地方，都会明确标记。

## 2. 总体结论

### 2.1 先用一句话说清模型在做什么

HRA-GNN 接收“一整张异构行为图”，把每个节点沿不同类型关系收到的消息分别编码，再
判断每种关系当前表现与训练期间学到的“正常关系原型”偏离多远；偏离越明显的关系越可能
获得更高注意力。所有节点经过两层这种关系感知更新后，再汇总成一个固定长度的图向量，
最后用“离正常中心有多远”和“看起来是否像未经扰动的原图”共同产生异常分数。

它做的是**图级异常检测**：

```text
输入：一张完整的调用图/行为图
输出：一个标量异常分数
```

它不是：

```text
给每个节点分类
给每条边分类
直接定位哪一条关系是人工标注的故障
```

关系注意力和关系偏差可以作为解释信号，但当前数据没有节点级或边级异常真值，不能把
attention 直接当作“故障关系标签”。

### 2.2 先区分四类容易混在一起的对象

| 对象 | 代码中的载体 | 含义 | 是否可训练 |
|---|---|---|---:|
| 节点特征 | `graph.x` | 每个节点的观测属性 | 否，输入数据 |
| 节点表示 | `h` / `node_embeddings` | 网络逐层学习的隐藏向量 | 是，由参数计算 |
| 关系原型 | `prototype[r]` | 某一层某种关系在正常训练图中的运行均值 | 否，EMA buffer |
| SVDD 中心 | `svdd_center` | 正常训练图在图表示空间中的固定中心 | 否，初始化后冻结 |

关系原型和 SVDD 中心不是同一个东西：

- 关系原型工作在**每层关系通道空间**，用于决定“节点该关注哪种关系”；
- SVDD 中心工作在**最终图向量空间**，用于决定“整张图离正常区域有多远”。

### 2.3 整条前向计算主干

对一个含 `N` 个节点、`E` 条边的 batch，完整主干是：

```text
GraphSample
  x                [N, F]
  node_type        [N]
  edge_index       [2, E]
  edge_type        [E]
  batch            [N]
        |
        v
按节点类型分别线性投影
  h^(0)            [N, H]
        |
        v
预计算每条边的关系三元组 ID
  relation_id      [E]
        |
        v
第 1 个 RelationDeviationLayer
  关系专属边消息          [E, H]
  同目标、同关系均值聚合  [P, H]
  与正常关系原型比较      [P]
  节点内跨关系 attention  [P]
  节点更新                [N, H]
        |
        v
第 2 个 RelationDeviationLayer
  重复上述过程，输出 h^(2) [N, H]
        |
        v
每张图分别做 max/mean pooling
  g_max, g_mean     [B, H]
        |
        v
逐维门控混合
  pooled            [B, H]
        |
        v
图投影
  embedding         [B, D]
        |
        +-----------------------------+
        |                             |
        v                             v
与 SVDD 中心计算距离             SSL 两层 MLP
  S_dist [B]                     logit [B]
        |                             |
        +--------------+--------------+
                       v
       S = S_dist * (1 + 1 - sigmoid(logit))
```

其中：

- `B`：当前 batch 的图数量；
- `F`：原始特征维数；
- `H`：节点隐藏维数；
- `D`：最终图表示维数；
- `P`：当前层实际出现的 `(目标节点, 关系)` 组合数量，不等于边数，也不等于完整关系数。

### 2.4 训练时不是一次前向，而是两次

每个训练 batch 共享同一个编码器做两次计算：

```text
                    同一个 HRA-GNN 参数
                    /               \
原始正常图 ----------> 原图前向       增强图 ----------> 增强图前向
                       |                                |
                       | 更新关系原型                   | 不更新关系原型
                       |                                |
                       +--> SVDD loss                   |
                       |                                |
                       +--> SSL logit=原图              +--> SSL logit=扰动图
                                    \                  /
                                     BCE 自监督损失
                                           |
                         total = SVDD + eta * SSL
                                           |
                                        backward
```

关键点：

1. 原始图和增强图的编码器参数完全共享；
2. 两次前向产生的梯度都会更新编码器参数；
3. 只有原始图允许更新关系原型；
4. 关系原型通过 EMA 更新，不接收梯度；
5. 数据集的正常/异常标签不进入 `_train_batch()` 的损失；
6. 训练索引已经由官方划分限定为正常图。

### 2.5 “无监督”在这里究竟是什么意思

这套实现更准确的称呼是：

```text
仅正常样本训练的 one-class / partially-unsupervised 图级异常检测
```

原因是：

- 训练损失不使用正常/异常类别标签；
- 但数据准备阶段需要知道哪些图是正常图，才能构造纯正常训练集；
- TraceLog 的验证集同时含正常和异常图，并使用验证 AUC 选择 checkpoint；
- 测试标签用于最后计算 AUC/AP。

因此不能笼统理解成“从头到尾完全看不到任何标签”。标签的使用边界是：

| 阶段 | 是否使用图级真值 | 用途 |
|---|---:|---|
| 官方数据划分 | 是 | 确保训练集为正常图 |
| 前向与反向传播 | 否 | 损失中没有真实类别项 |
| TraceLog 早停 | 是 | 计算验证 AUC 并选择 checkpoint |
| FlowGraph 早停 | 否 | 没有验证集，监控负训练损失 |
| 最终测试 | 是 | 只计算 AUC/AP，不反向传播 |

### 2.6 数据中的标签到底是什么

当前实现只有**图级标签**：

```text
label = 0：正常图
label = 1：异常图
```

发布数据的 `trace_bool=true` 表示该 trace 正常，因此加载时映射为 0。其余值映射为 1。

没有以下监督：

- 节点是否异常；
- 边是否异常；
- 哪种关系导致异常；
- 异常开始和结束位置；
- 具体故障根因。

自监督分支中的 1/0 也不是数据集真值：

```text
原图 SSL 伪标签 = 1
增强图 SSL 伪标签 = 0
```

它表示“是否像未经人工扰动的原图”，不是“正常/异常”的真实标签。

### 2.8 当前实现覆盖范围

当前实现覆盖了论文描述的主要训练链路：

```text
异构图
  -> 类型专属输入映射
  -> 关系三元组编码
  -> 关系专属消息变换
  -> 关系原型与标准化偏差
  -> 偏差调制关系注意力
  -> 多层节点编码
  -> 最大/均值门控混合读出
  -> DeepSVDD + 原图/增强图自监督训练
  -> 论文乘法异常评分
```

核心模型不是对原 HRGCN 文件做局部修改，而是在 `hra_gnn/` 中重新实现，原代码
保留在 `src/` 供对照。

必须特别注意：

- HRA-GNN 主体机制有代码对应；
- FlowGraph 发布数据与论文的边类型描述不一致；
- 首批次固定 SVDD 中心导致严重随机种子敏感性；
- HGT、DeepTraLog、GLocalKD 是统一协议下的适配器，不是官方源码复现；
- 当前五种子结果不能稳定复现论文的 FlowGraph 1.0。

## 4. 输入图与任务定义

### 4.1 单图数据结构

`GraphSample` 位于 `hra_gnn/graph.py`，包含：

```text
x             [N, F]    节点特征
edge_index    [2, E]    源节点和目标节点索引
node_type     [N]       每个节点的类型
edge_type     [E]       每条边的类型
edge_weight   [E]       可选边权
label         标量      0 为正常，1 为异常
batch         [N]       批处理时每个节点所属的图
```

标签转换在 `data.py::_boolean_label()` 中完成。发布数据中 `trace_bool=true`
表示正常，因此转换为 0；其他值转换为异常 1。

### 4.2 任务

训练阶段只使用官方训练划分。该任务按论文设定为无监督或单类图级异常检测：

- 不使用测试异常标签反向传播。
- DeepSVDD 让训练图表示靠近固定中心。
- SSL 使用人工增强图作为伪负样本。
- 测试阶段根据异常分数排序，计算 AUC 和 AP。

### 4.3 官方数据划分

划分加载在 `data.py::load_provided_splits()`：

- TraceLog：训练、验证、测试三份图 ID 文件。
- FlowGraph：训练文件和评估文件，没有独立验证集。

FlowGraph 没有验证集时，`trainer.py::train()` 使用负训练损失作为早停监控量。
这一行为虽然保证代码可运行，但不能等价于用独立验证集选择 checkpoint。

### 4.4 四个原始文件如何组成一张图

`CSVGraphDataset` 不会把一行 CSV 当成一张图。它需要把四份信息按 `trace_id` 合并：

```text
node_feature_norm.csv
  trace_id, node_id, feature_1, ..., feature_F
                    |
                    +--> x [N, F]

edge_index.csv
  trace_id, src_id, dst_id, edge_type?, weight?
                    |
                    +--> edge_index [2, E]
                    +--> edge_type [E]
                    +--> edge_weight [E] 或 None

node_types.txt
  每行记录一张图中各类型包含的节点编号
                    |
                    +--> node_type [N]

trace_info.csv
  trace_id, trace_bool, ...
                    |
                    +--> graph_id
                    +--> label
```

加载过程位于 `data.py::CSVGraphDataset.__init__()` 和 `graph_at_id()`：

1. 节点表按 `(trace_id, node_id)` 稳定排序；
2. 边表按 `trace_id` 排序；
3. `_group_slices()` 预计算每张图在连续 NumPy 数组中的切片；
4. 访问某张图时，用切片构造 tensor；
5. `node_types.txt` 中第 `type_id` 组的节点被赋值为该类型；
6. 最终构造 `GraphSample`，其构造函数立即检查形状。

这样做的原因不是模型理论，而是性能：如果每个 epoch 对 132,485 张 TraceLog 图反复
执行 pandas 条件筛选，CPU 数据准备会成为瓶颈。现在只在初始化时排序和建立索引。

### 4.5 edge type 缺失时发生什么

`data.py` 中：

```python
self._edge_type_values = (
    edges["edge_type"].to_numpy(dtype=np.int64).copy()
    if "edge_type" in edges
    else np.zeros(len(edges), dtype=np.int64)
)
```

因此如果 `edge_index.csv` 没有 `edge_type` 列，所有边都被设为类型 0。

这正是当前 FlowGraph 的情况。其 26 个 event code 位于节点特征列，不是边类型。模型
仍可利用源/目标节点类型形成 `8 * 1 * 8 = 64` 种关系三元组，但不能声称使用了 26 种
显式边关系。

### 4.6 标签在哪里用，在哪里绝对不能用

训练器收到的不是“全数据集然后自行看标签训练”，而是索引列表：

```python
self.splits["train"]
self.splits["validation"]
self.splits["test"]
```

`_train_batch(indices)` 只按索引取图，从未读取 `graph.label` 来计算损失。标签只在
`evaluate()` 中进入 `labels` 列表，最后交给 `roc_auc_score` 和
`average_precision_score`。

从代码审查角度，应区分：

```text
用标签构造实验划分       数据协议行为
用标签选择 checkpoint    模型选择行为
用标签计算最终指标       评估行为
把标签放进 loss.backward  监督训练行为
```

当前实现前三种都有，第四种没有。

### 4.7 为什么不需要人工重新标注

TraceLog 和 FlowGraph 已提供图级 `trace_bool`。复现工作不是重新让人逐图标注，而是
遵循发布数据已有真值：

```text
trace_bool=true / normal / benign / 字符串"1" -> label 0
其余值                                      -> label 1
```

需要警惕字符串和数值的语义不同：

- 字符串 `"1"` 被明确按正常处理；
- 数值 `1` 的 `bool(1)` 也按正常处理；
- 数值 `0` 按异常处理。

如果接入新数据，不能默认新数据的 `1` 仍表示正常。必须先在 loader 层统一语义，否则
AUC 方向会整体翻转。

### 4.8 多图 batch 不是把图连起来

`batch_graphs()` 将多张图拼成一张**断连大图**。例如：

```text
图 A：3 个节点，边 0 -> 1
图 B：2 个节点，边 0 -> 1
```

合批后：

```text
图 A 节点仍为 0,1,2
图 B 节点平移为 3,4，边变成 3 -> 4
batch = [0,0,0,1,1]
```

没有 `2 -> 3` 等跨图边。消息传递只依赖 `edge_index`，读出则依赖 `batch` 把节点重新
分组。因此可以用一次大张量运算处理多张图，同时保持数学独立。

## 5. 核心创新一：异构关系三元组

### 5.1 论文意图

论文不是只按“边类型”区分关系，而是使用：

```text
r = (源节点类型, 边类型, 目标节点类型)
```

例如：

```text
(服务, 调用, 服务)
(进程, 写入, 文件)
```

即使边类型相同，只要源或目标节点类型不同，也视为不同关系。

### 5.2 代码实现

实现位于 `graph.py::canonical_relation_ids()`：

```python
relation_id = (
    source_node_type * num_edge_types + edge_type
) * num_node_types + destination_node_type
```

若节点类型数为 `T_v`、边类型数为 `T_e`，关系总空间大小为：

```text
R = T_v * T_e * T_v
```

对应 `model.py::HRAGNN.__init__()` 中：

```python
self.num_relations = num_node_types * num_edge_types * num_node_types
```

### 5.3 审查要点

- 编号公式是一一映射，不会让不同三元组碰撞。
- 未在某张图中出现的关系不会参与该图消息聚合。
- 模型仍会为完整关系空间分配参数和原型缓冲区。
- FlowGraph 当前只有 1 种显式边类型，无法按论文表述构造 26 种边关系。

### 5.4 为什么只用 edge type 不够

假设边类型都叫“访问”，以下两条边的业务语义仍可能完全不同：

```text
服务 --访问--> 数据库
用户 --访问--> 页面
```

如果只按 edge type 建模，两类交互共享同一个变换矩阵和原型；数据库访问异常可能被大量
正常页面访问稀释。三元组关系把“谁通过什么关系连接谁”作为最小语义单元：

```text
(服务, 访问, 数据库) != (用户, 访问, 页面)
```

代价是关系空间按 `T_v^2 * T_e` 增长。TraceLog 的 256 种完整关系、隐藏维 128 下，
单层 `relation_weight` 参数量为：

```text
256 * 128 * 128 = 4,194,304
```

两层仅关系矩阵就约 839 万参数，这也是 TraceLog 模型总参数达到 8,570,625 的主要原因。
FlowGraph 因 `R=64、H=32`，总参数只有 149,185。

这两个参数量按**配置中的完整关系空间**分配，不是按实际出现的关系动态分配。如果
TraceLog 全数据确实只使用部分节点/边类型，大量关系矩阵将永远收不到梯度。复杂度和
参数效率实验应同时报告“理论关系数”和“实际出现关系数”。

## 6. 核心创新二：关系专属消息传递

### 6.1 类型专属输入映射

不同节点类型使用不同输入线性层：

```python
self.type_projection = nn.ModuleList(
    [nn.Linear(input_dim, hidden_dim) for _ in range(num_node_types)]
)
```

对类型 `t` 的节点：

```text
h_v^(0) = LeakyReLU(W_t x_v + b_t)
```

实现位于 `model.py::HRAGNN.encode_nodes()`。

当前发布数据已经将所有节点特征处理为统一列数，因此每个类型投影层的输入维度
相同。如果原始异构图的不同节点类型具有不同特征维度，则当前数据接口需要增加
按类型独立的输入维度配置。

### 6.2 关系专属变换

每层为每个关系分配独立矩阵：

```text
W_msg[r] 的形状为 [H, H]
全部参数形状为 [R, H, H]
```

对边 `u -> v`：

```text
m_(u,v,r) = h_u W_msg[r]
```

代码使用批量矩阵乘法：

```python
transformed = torch.bmm(
    h[source].unsqueeze(1),
    self.relation_weight[relation_ids],
).squeeze(1)
```

### 6.3 按“关系-目标节点”聚合

同一目标节点、同一关系的消息取均值：

```text
h_(v,r) = mean({m_(u,v,r) | (u,v) 属于关系 r})
```

工程实现将 `(relation_id, destination)` 编成唯一整数：

```python
relation_node_id = relation_id * num_nodes + destination
```

随后通过 `torch.unique(..., return_inverse=True)` 和 `index_add_` 一次完成所有
关系的聚合。

这与逐关系循环执行均值聚合在数学上等价，但显著减少 GPU kernel 启动次数。

### 6.4 残差和归一化

每层最终输出：

```text
h_new = LayerNorm(
    h_old + Dropout(LeakyReLU(root(h_old) + fused_relation_message))
)
```

实现位于 `RelationDeviationLayer.forward()` 末尾。

需要审查论文是否明确包含：

- `root` 线性变换；
- 外层残差；
- LayerNorm；
- Dropout 的具体位置。

这些细节会影响数值结果。如果原论文公式没有明确给出，应在论文方法或实验设置
中补充，而不能只存在于代码。

### 6.5 一层 `RelationDeviationLayer` 的完整计算顺序

下面按 `forward()` 的真实执行顺序展开。输入为：

```text
h               [N, H]
edge_index      [2, E]
relation_ids    [E]
```

#### 第 1 步：保留节点自身信息

```python
base = self.root(h)
```

得到：

```text
base [N, H]
```

这条 root 分支不经过邻居，即使某节点没有入边，也有可更新的自身表示。

#### 第 2 步：按边取源节点

```python
source, destination = graph.edge_index
h_source = h[source]
```

`h_source` 形状为 `[E, H]`。同一个源节点有多条出边时，其表示会重复出现多次。

#### 第 3 步：为每条边选关系矩阵并变换

```python
W_edge = self.relation_weight[relation_ids]  # [E, H, H]
transformed = bmm(h_source[:, None, :], W_edge).squeeze(1)
```

输出 `transformed [E, H]`。这里每条边不是都有独立参数，而是关系 ID 相同的边共享
`W_r`。

若存在 `edge_weight`：

```text
transformed[e] *= edge_weight[e]
```

#### 第 4 步：把“同目标节点、同关系”的边归为一组

```text
pair_id[e] = relation_id[e] * N + destination[e]
```

这个整数同时编码关系和目标节点。`torch.unique()` 得到所有实际出现的 pair：

```text
unique_pairs       [P]
inverse            [E]
```

`inverse[e]` 表示第 `e` 条边属于哪一个 pair。

#### 第 5 步：同组边消息取均值

```text
sum[p]   = Σ transformed[e]
count[p] = 组内边数
relation_values[p] = sum[p] / count[p]
```

得到：

```text
relation_values    [P, H]
active_nodes       [P]
active_relations   [P]
```

此时一条记录的含义是：

```text
目标节点 v 在关系 r 下收到的聚合消息 h_(v,r)
```

#### 第 6 步：计算每个关系通道偏离正常原型的程度

```text
standardized[p] =
    (relation_values[p] - prototype[r]) / prototype_scale[r]

deviation[p] =
    sqrt(mean_h(standardized[p]^2) + 1e-8)
```

若关系尚未初始化，当前 deviation 被强制为 0。

#### 第 7 步：收集本 batch 的原型统计

仅当：

```text
self.training == True
并且 update_prototypes == True
```

才收集 `relation_values`。收集时会 `detach().clone()`，所以原型更新不进入计算图。

#### 第 8 步：关系融合

完整模型：

```text
semantic[p] = MLP([h[v] || relation_values[p]])
score[p] = semantic[p] + gamma * deviation[p]
attention[p] = softmax(score[p] / temperature)
```

softmax 只在同一个 `active_node` 内进行。然后：

```text
fused_messages[v] =
    Σ_{p: active_node[p]=v} attention[p] * relation_values[p]
```

#### 第 9 步：节点更新

```text
fused = base + fused_messages
h_new = LayerNorm(h + Dropout(LeakyReLU(fused)))
```

注意存在两层“自身信息”：

1. `base = root(h)` 位于激活内部；
2. 最外层还有 `h + ...` 残差。

### 6.6 用一个小图手算关系分组

设节点类型数 `T_v=3`，边类型数 `T_e=2`，有 4 个节点：

```text
节点 0：服务，type=0
节点 1：进程，type=1
节点 2：文件，type=2
节点 3：文件，type=2
```

有三条边：

```text
0 --edge_type 0--> 1
1 --edge_type 1--> 2
3 --edge_type 1--> 2
```

关系 ID：

```text
r(0,0,1) = (0 * 2 + 0) * 3 + 1 = 1
r(1,1,2) = (1 * 2 + 1) * 3 + 2 = 11
r(2,1,2) = (2 * 2 + 1) * 3 + 2 = 17
```

后两条边虽然 edge type 相同、目标类型相同，但源类型不同，因此不是同一关系。如果第三条
边改成 `1 -> 2` 的另一条平行观测，它才会与第二条边同属 `(目标节点2, 关系11)`，两条
消息在进入 attention 之前先取均值。

当 `N=4` 时，第二条边的 pair ID：

```text
11 * 4 + 2 = 46
```

从 `46` 可反解：

```text
target = 46 % 4 = 2
relation = 46 // 4 = 11
```

这就是代码用一个整数完成二维分组的原因。

### 6.7 两层网络到底多看了多远

第一层中，节点 `v` 融合直接入邻居的信息；第二层的源节点表示已经包含其直接邻居信息，
因此节点 `v` 最多可接收二跳结构信号。简化表示：

```text
h_v^(0) = 类型投影(x_v)
h_v^(1) = Layer1(h^(0), edges, relations)
h_v^(2) = Layer2(h^(1), edges, relations)
```

两层各自拥有独立的：

- `relation_weight`；
- 关系原型和尺度；
- 语义注意力 MLP；
- root、LayerNorm。

所以“关系 11 在第一层的原型”和“关系 11 在第二层的原型”是两个不同统计量。前者描述
较低层局部特征，后者描述经过一轮传播后的高阶交互特征。

## 7. 核心创新三：关系正常原型

### 7.1 保存的统计量

每一层、每一种关系维护：

```text
prototype[r]             [H]  关系正常均值
prototype_scale[r]       [H]  关系逐维标准差
prototype_initialized[r]      是否初始化
prototype_updates[r]          更新次数
```

它们通过 `register_buffer(..., persistent=True)` 注册：

- 会随 checkpoint 保存。
- 不由梯度更新。
- 不属于可训练参数。

### 7.2 初次初始化

对当前 batch 中关系 `r` 的全部关系通道表示 `z_r`：

```text
mu_batch[r] = mean(z_r)
sigma_batch[r] = std(z_r)
```

第一次出现该关系时：

```text
prototype[r] = mu_batch[r]
scale[r] = max(sigma_batch[r], min_scale)
```

### 7.3 指数滑动平均

后续 batch 使用动量 `m`：

```text
prototype[r] =
    m * prototype[r] + (1 - m) * mu_batch[r]

scale[r] =
    m * scale[r] + (1 - m) * sigma_batch[r]
```

默认 `m=0.9`，默认最小尺度 `1e-3`。

实现位于 `model.py::RelationDeviationLayer._update_prototype()`。

### 7.4 为什么要按 batch 提交

训练器在原始图前向前调用：

```python
model.begin_prototype_batch()
```

所有关系表示先暂存，前向结束后调用：

```python
model.commit_prototype_batch()
```

这样每层每种关系每个 batch 只更新一次，符合论文“基于当前批次统计量更新”的
表述。测试 `test_prototypes_commit_once_per_batch()` 专门验证这一点。

### 7.5 增强图不能更新原型

原始图：

```python
self.model(graph, update_prototypes=True)
```

增强图：

```python
self.model(augmented, update_prototypes=False)
```

因此人工扰动不会直接污染正常原型。

### 7.6 仍存在的污染风险

虽然增强图不更新原型，但论文假设训练集“以正常图为主”。如果训练集中包含真实
异常图，这些异常图仍会参与 EMA 更新，导致原型污染。

当前代码没有：

- 截断异常关系表示；
- 中位数或分位数鲁棒估计；
- 按异常分数过滤训练样本；
- 置信度加权更新；
- 独立的原型预热阶段。

这正是专家所说“需验证关系原型对异常样本的鲁棒性”的代码依据。

### 7.7 原型更新与当前 batch 前向的先后关系

一个容易误解的地方是：当前 batch 收集了原型统计，不代表当前 batch 立即使用更新后的
原型。真实时序是：

```text
batch k 开始
  |
  +-- 使用 batch k-1 结束时的 prototype 计算 deviation
  |
  +-- 暂存 batch k 的 relation_values
  |
batch k 原图前向结束
  |
  +-- commit：用 batch k 统计更新 prototype
  |
  +-- 后续增强图前向看到更新后的 prototype，但 update_prototypes=False
  |
batch k+1 原图前向
  |
  +-- 使用 batch k 更新后的 prototype
```

第一批原图前向时所有原型未初始化，所以偏差全部为 0；第一批 commit 后才得到初始原型。

### 7.8 原型为什么是 buffer 而不是 Parameter

如果把原型设为 `nn.Parameter`，它会被优化器朝“降低当前训练损失”的方向更新，不再是
清晰的正常统计中心。当前设计使用：

```python
register_buffer(..., persistent=True)
```

带来三点性质：

1. `model.to(device)` 时自动移动到 GPU；
2. 保存 checkpoint 时进入 `state_dict`；
3. 不出现在 `model.parameters()`，Adam 不会更新它。

EMA 的含义是“运行统计”，类似 BatchNorm 的 running mean，而不是一个普通神经网络权重。

### 7.9 哪些量有梯度，哪些没有

```text
relation_values ------------------------------ 有梯度
      | \
      |  \--> semantic score ----------------- 有梯度
      |
      +----> deviation ------------------------ 对 relation_values 有梯度
                  |
prototype / scale ---------------------------- 无梯度、detach
```

因此偏差调制仍能训练关系矩阵：增大或减小 `relation_values` 会改变 deviation 和最终
attention，梯度会回到 `relation_weight` 及前一层节点表示。只是模型不能通过反向传播
“作弊式移动”原型来降低偏差。

### 7.10 为什么使用均值和标准差

只比较欧氏距离会让天然尺度大的关系总是看起来更异常。例如某关系某一维正常波动为 100，
另一关系正常波动为 0.01，直接距离不可比较。标准化后：

```text
偏离 1 个自身标准差
```

在不同关系和维度上具有近似统一的含义。当前偏差是各隐藏维标准化平方的均值再开根号，
可理解为逐维 z-score 的 RMS。

但该设计隐含近似单峰、尺度可由标准差描述。若正常关系本身多模态，单原型均值可能落在
两个正常簇中间，反而把正常样本判得很远。这也是需要多原型或鲁棒统计实验的原因。

## 8. 核心创新四：标准化关系偏差

### 8.1 数学定义

对节点 `v` 在关系 `r` 下的通道表示 `h_(v,r)`：

```text
z_(v,r) =
    (h_(v,r) - prototype[r]) / max(scale[r], epsilon)

d_(v,r) =
    sqrt(mean(z_(v,r)^2) + 1e-8)
```

代码位于 `RelationDeviationLayer.forward()` 中 `standardized` 和
`deviations` 的计算。

这里的 `mean` 是对隐藏维求均值，因此偏差是标量。

### 8.2 未初始化关系

某关系原型尚未初始化时：

```text
d_(v,r) = 0
```

因此第一批数据的偏差调制完全不生效，只依赖语义注意力。第一批结束后才建立
原型。

### 8.3 数值稳定性

- `scale` 至少为 `prototype_min_scale`。
- 平方均值后加 `1e-8` 再开根号，避免零点梯度出现 NaN。
- 原型和尺度在偏差计算中不参与梯度。

### 8.4 偏差只表示“大小”，不表示“方向”

平方和开根号会丢失正负方向：

```text
比原型高 3 个标准差
比原型低 3 个标准差
```

二者 deviation 相同。这样设计适合回答“偏离是否显著”，但不能区分两种偏离的业务含义。
如果一种方向对应故障、另一种方向是正常负载下降，模型仍需依赖 `semantic_score` 和其他
关系上下文区分。

## 9. 核心创新五：关系偏差调制注意力

### 9.1 基础语义分数

对每个有效的 `(节点, 关系)` 通道：

```text
s_sem(v,r) =
    MLP([h_v || h_(v,r)])
```

MLP 为：

```text
Linear(2H, H)
-> LeakyReLU
-> Linear(H, 1, 无偏置)
```

### 9.2 偏差调制

完整模式：

```text
s(v,r) = s_sem(v,r) + gamma * d_(v,r)
```

其中 `gamma` 对应 YAML 中：

```yaml
model:
  deviation_weight: 1.0
```

### 9.3 节点内关系 softmax

对同一目标节点的全部有效关系归一化：

```text
alpha_(v,r) =
    softmax_r(s(v,r) / temperature)
```

代码通过 `segment_softmax(scores, active_nodes, num_nodes)` 实现。分段 ID 是
目标节点编号，因此不同节点之间不会互相归一化。

### 9.4 跨关系融合

```text
message_v = sum_r alpha_(v,r) * h_(v,r)
```

代码通过：

```python
fused_messages.index_add_(
    0,
    active_nodes,
    attention.unsqueeze(1) * relation_values,
)
```

### 9.5 三种关系融合模式

```yaml
model:
  relation_fusion: deviation_attention
```

支持：

| 模式 | 含义 |
|---|---|
| `deviation_attention` | 语义分数加标准化偏差 |
| `semantic_attention` | 只有语义注意力，用于去除偏差调制 |
| `static_concat` | 静态关系通道线性投影，用于 HRGCN/拼接消融 |

`static_concat` 没有真的构造 `[节点, 关系数 * H]` 巨大张量。实现为：

```text
sum_r h_(v,r) W_concat[r]
```

这等价于“先按关系拼接，再乘一个按关系分块的大线性矩阵”，同时避免大量零填充。
这是数学等价的工程优化，不是另一种注意力机制。

### 9.6 为什么采用“加性调制”

基础语义注意力回答：

```text
这种关系的内容与当前目标节点是否匹配？
```

关系偏差回答：

```text
这种关系相对于训练期正常模式是否罕见？
```

两者相加后，模型可以同时利用任务学习出的语义分数和显式正常性统计：

```text
score = learned_semantic_score + gamma * statistical_deviation
```

若只使用偏差，任何罕见关系都可能被放大；若只使用语义注意力，训练集全是正常图时，
模型没有直接理由关注“偏离正常原型”这一异常检测特征。

### 9.7 `gamma` 和 temperature 分别控制什么

```text
gamma 增大：
  偏差在排序中的影响更强

temperature 减小：
  softmax 更尖锐，更接近只选最高分关系

temperature 增大：
  softmax 更平滑，更接近平均关系
```

两者不能互相替代。`gamma` 改变语义项与偏差项的相对比例；temperature 缩放二者相加后的
整体分数。

### 9.8 注意力机制什么时候实际不起作用

对某个目标节点，如果只有一种有效关系：

```text
softmax([score]) = [1]
```

此时无论偏差多大，关系融合结果都相同。偏差调制只有在一个目标节点同时收到两种或更多
关系通道时，才能改变跨关系权重。

这对解释 FlowGraph 很重要：虽然 `T_v=8` 仍可形成多种源/目标类型三元组，但显式边类型
只有一种。如果大部分节点只有单一入关系，核心 attention 的作用空间会显著缩小。

### 9.9 attention 不是因果解释

导出的 `mean_attention` 说明模型在当前前向中分配了多大权重，不能单独证明：

- 该关系真实导致了异常；
- 删除该关系一定降低异常分数；
- 该权重在不同 seed 中稳定。

因此专家要求的解释实验至少需要补：

1. 正常/异常图 attention 和 deviation 分布；
2. top-k 关系删除或保留后的分数变化；
3. 不同 seed 的关系排序一致性；
4. 成功案例与失败案例。

## 10. 核心创新六：门控混合图级读出

### 10.1 两类统计

对每张图的节点表示：

```text
g_max  = max_v h_v
g_mean = mean_v h_v
```

批处理中使用 `segment_max()` 和 `segment_mean()`，确保每张图独立池化。

### 10.2 门控

```text
gate = sigmoid(
    Linear(
        LeakyReLU(
            Linear([g_max || g_mean])
        )
    )
)
```

门向量形状为 `[H]`，不是单个标量。

### 10.3 混合

```text
g = gate * g_max + (1 - gate) * g_mean
```

含义：

- `g_max` 偏向捕捉少量节点上的局部强异常。
- `g_mean` 偏向捕捉全图范围的弱分布漂移。
- 每个隐藏维可独立选择更依赖最大值还是均值。

最后通过 `graph_projection` 得到图级表示。

### 10.4 消融

```yaml
model:
  readout: hybrid  # 或 max、mean
```

论文消融配置会分别运行：

- 仅最大池化；
- 仅均值池化；
- 完整门控混合读出。

### 10.5 为什么门是 `[H]` 而不是一个标量

若使用单标量门：

```text
整张图只能整体选择 max 或 mean
```

当前实现生成逐隐藏维门：

```text
gate[b, j] ∈ [0,1]
```

同一张图可以在第 5 个隐藏维主要采用 max，在第 23 个隐藏维主要采用 mean。因为不同隐藏
维可能编码不同关系模式，这比一个全局标量更灵活。

例如：

```text
gate = [0.9, 0.1]
g_max  = [8.0, 2.0]
g_mean = [1.0, 5.0]

pooled[0] = 0.9*8.0 + 0.1*1.0 = 7.3
pooled[1] = 0.1*2.0 + 0.9*5.0 = 4.7
```

第一维保留局部极值，第二维保留整体统计。

### 10.6 门控读出为何与异常检测问题相关

局部强异常可能只影响一两个节点。若图有 8,000 个节点，均值池化会把单节点异常除以
8,000；最大池化更容易保留它。

全局弱异常可能让很多节点都轻微偏移。单个节点不极端，max 不稳定，而 mean 能积累整体
变化。

门控的设计动机是合理的，但目前代码不会显式知道某张图属于“局部异常”还是“全局异常”。
它只能通过最终联合损失间接学习门。因此必须用 `max-only / mean-only / hybrid` 消融和
按异常类型分组结果验证，不能仅凭直觉宣称已经解决两类异常。

### 10.7 从节点表示到最终 embedding 的形状

TraceLog batch size 为 8 时：

```text
最终节点表示         [N_total, 128]
segment_max          [8, 128]
segment_mean         [8, 128]
concat               [8, 256]
gate                 [8, 128]
pooled               [8, 128]
graph_projection     [8, 128]
ssl_head             [8]
```

FlowGraph batch size 为 4 时相同流程，只是隐藏维和输出维均为 32。

## 11. DeepSVDD 单类学习

### 11.1 中心初始化

第一次训练 batch 前向后：

```text
c = mean({g_i | i 属于首批次})
```

中心注册为 buffer，初始化后固定，不参与梯度。

绝对值小于 `1e-4` 的中心分量被改为 `1e-4`，用于降低全零坍缩风险。

### 11.2 SVDD 损失

```text
L_svdd = mean((g_i - c)^2)
```

代码对 batch 维和隐藏维整体取均值。

论文通常写成平方 L2 距离求和。当前实现除以固定隐藏维数，只改变损失和分数的
整体尺度；对固定维度下的 AUC/AP 排序不产生影响，但会影响与 SSL 损失的相对
量级，因此 `ssl.loss_weight` 与该尺度绑定。

### 11.3 重大不稳定来源

中心只由首批次估计。不同随机种子会改变：

- 首批次包含哪些训练图；
- 初始网络权重；
- 首批次图表示；
- 最终固定中心。

FlowGraph 五种子 AUC 从 0.0308 到 0.9837，说明该策略并不稳健。建议增加：

1. 全训练集预扫描初始化中心；
2. 多批次预热均值；
3. 指数更新中心后再冻结；
4. 中位数或截断均值中心；
5. 首批次组成敏感性实验。

这些属于改进实验，不能在不说明的情况下替换论文默认设置。

### 11.4 为什么 SVDD 不需要异常标签

训练集只含正常图，目标不是学习“正常类和异常类之间的分类边界”，而是让所有正常图表示
靠近一个中心：

```text
正常图 g_i -> 尽量靠近 c
```

测试时没有再训练分类器，直接把距离当异常程度：

```text
离 c 越远 -> 越异常
```

这要求训练集确实能代表正常分布，也要求中心估计稳定。若中心本身偏了，整个距离排序都会
偏，这正是 FlowGraph seed 55 AUC 接近 0 的机制性风险。

### 11.5 第一批上的梯度并不恒为零

中心初始化为首批 embedding 的均值并 `detach`：

```text
c = mean(g_1, ..., g_B)
```

随后同一批计算：

```text
mean_i,j (g_i[j] - c[j])^2
```

当 batch 内图表示不完全相同时，各图仍有非零梯度。如果 batch size 为 1，并且忽略中心
近零分量被强制改成 `1e-4` 的修正，中心等于该图本身，首批 SVDD loss 为 0；实际代码
可能因该修正留下很小的非零损失。当前 TraceLog/FlowGraph batch size 分别为 8/4，因此
首批通常有明确的非零梯度。

### 11.6 为什么 mean 距离与论文常见 sum 距离有差别

代码：

```python
(embedding - center).square().mean(dim=-1)
```

论文常见写法：

```text
||embedding - center||_2^2 = sum_j(...)
```

对固定输出维 `D`，两者只差常数 `D`，所以只用 SVDD 排序时 AUC 不变。但联合训练中：

```text
L_total = L_svdd + eta * L_ssl
```

这个常数会改变两项损失的相对强度。因此不能把 `eta` 脱离具体距离实现直接搬到另一份代码。

## 12. 自监督增强分支

### 12.1 伪标签定义

共享编码器分别处理：

- 原始图：伪标签 1；
- 增强图：伪标签 0。

图表示经过两层 MLP 输出一个 logit：

```text
p = sigmoid(ssl_head(g))
```

### 12.2 自监督损失

```text
L_ssl = BCEWithLogits(original_logits, 1)
      + BCEWithLogits(augmented_logits, 0)
```

实际代码将两部分拼接后统一求均值。

### 12.3 联合训练目标

```text
L = L_svdd + eta * L_ssl
```

`eta` 位于：

```yaml
ssl:
  loss_weight: ...
```

FlowGraph 和 TraceLog 使用不同权重，具体值见各自 YAML。

### 12.4 需要严格质疑的认知假设

增强图不是已知真实异常，只是人工扰动。把它设为伪负样本隐含假设：

```text
人工扰动方向 ≈ 真实异常方向
```

该假设未必成立。当前五种子结果中 SSL 单独 AUC 波动明显，说明 SSL 分支可能
学习到增强伪影，而不是实际异常机制。

专家要求“分析不同自监督增强策略的独立作用”，应分别运行：

- 仅边扰动；
- 仅边添加；
- 仅节点类型交换；
- 仅边类型交换；
- 两两组合；
- 全部增强；
- 完全关闭 SSL。

### 12.5 SSL 伪标签与真实标签不要混淆

假设训练 batch 中有 4 张正常图：

```text
原图 logits       [4]，目标 [1,1,1,1]
增强图 logits     [4]，目标 [0,0,0,0]
拼接 logits       [8]
```

BCE 训练的是：

```text
“这张图像原图还是人工扰动图？”
```

而不是：

```text
“这张图是真实正常还是实际故障？”
```

测试时将：

```text
p_original = sigmoid(logit)
S_ssl = 1 - p_original
```

作为辅助异常信号，是把“像扰动图”近似解释为“可能异常”。这一近似必须通过不同增强独立
消融和真实故障类型分析验证。

### 12.6 两条分支的梯度如何汇合

```text
原图 embedding --> SVDD loss -----------------------+
       |                                             |
       +--> original SSL logit --> BCE --------------+--> total loss
                                                     |
增强图 embedding --> augmented SSL logit --> BCE ----+
```

`total_loss.backward()` 后：

- 图投影层同时收到 SVDD 和 SSL 梯度；
- 关系层同时收到原图、增强图两次前向的梯度；
- `ssl_head` 只收到 BCE 梯度；
- `svdd_center` 不收到梯度；
- 关系 prototype/scale 不收到梯度；
- 增强操作本身在 CPU 上离散执行，不可微，也不需要梯度。

### 12.7 论文乘法评分中 SSL 的实际影响范围

正式评分：

```text
S = S_dist * (1 + S_ssl)
S_ssl = 1 - sigmoid(logit) ∈ [0,1]
```

因此：

```text
1 + S_ssl ∈ [1,2]
```

SSL 最多把某张图的 SVDD 距离放大两倍，不能像加权和那样独立主导结果。如果 SVDD 排序
整体反向，SSL 通常无法完全修复。

FlowGraph seed 55 是直接例子：

```text
SVDD AUC 约 0.0308
SSL AUC  约 0.9666
最终乘法 AUC仍约 0.0308
```

这说明最终结果主要受 SVDD 中心和距离控制，不能看到 SSL 单项很高就认为联合模型稳定。

### 12.8 同一训练 batch 中两次前向存在原型版本差异

当前训练顺序是：

```text
旧原型 P_t
  -> 原图前向，得到 z_original
  -> 用原图统计提交 EMA，得到新原型 P_(t+1)
  -> 增强图前向，得到 z_augmented
```

因此，原图前向使用 `P_t`，增强图前向使用 `P_(t+1)`。两条 SSL 分支不仅输入图
不同，原型状态也略有不同。这个差异在原型变化较快的训练早期尤其明显。

这不是标签泄漏，因为更新仍只来自原图；但它是一个潜在混杂变量：SSL 分类器
可能部分利用原型版本变化造成的表示差异，而不是只学习“原图与增强图”的差异。

更严格的对照实现应提供以下模式并做消融：

1. **共享旧原型**：原图和增强图都使用 `P_t`，两次前向结束后再提交原图统计；
2. **当前模式**：原图使用 `P_t`，增强图使用 `P_(t+1)`；
3. **冻结原型**：预热后固定原型，检查 SSL 效果是否仍存在。

其中模式 1 的因果解释最干净，因为两条分支唯一的系统性差异是图增强。当前代码
实现的是模式 2，论文和实验报告中应如实说明。

## 13. 四种异构图增强

实现位于 `hra_gnn/augment.py`。

### 13.1 边扰动

随机选择一定比例的边，保持目标节点类型不变，在同类型候选节点中替换目标。

保持：

- 源节点不变；
- 边类型不变；
- 目标节点类型不变。

改变：

- 具体目标节点。

### 13.2 边添加

随机选择已有边作为关系模板，然后：

- 从模板源节点类型中随机选新源节点；
- 从模板目标节点类型中随机选新目标节点；
- 复制模板边类型。

因此新边的关系三元组来自图中已观察关系。

### 13.3 节点类型交换

随机选择偶数个节点，两两交换节点类型。完成后检查所有边产生的关系三元组是否
仍属于增强前观察到的关系集合。

最多尝试 20 次；始终不合法时返回原图副本。

### 13.4 边类型交换

随机选择偶数条边，两两交换边类型，同样检查关系三元组是否仍属于原图观察集合。

FlowGraph 只有一种显式边类型，因此该增强在 FlowGraph 上没有作用。

### 13.5 工程优化

早期实现对每条边重复执行 `nonzero`，FlowGraph 增强成为主要瓶颈。当前实现：

- 预先按类型建立候选集合；
- 一次生成全部模板；
- 使用张量批量赋值；
- 只对少量类型分组循环。

采样含义不变，但随机数消费顺序与逐元素版本不完全相同。因此，同一随机种子不
保证与早期开发版本产生逐边完全一致的增强图。

### 13.6 每张图每次只随机选择一种增强

默认配置列出四种方法，不表示四种操作会按顺序全部施加到同一张图。实际入口：

```python
method = method or self.random.choice(self.methods)
```

所以每次 `augment(graph)`：

1. 从允许的方法中均匀随机选一种；
2. 只执行该方法；
3. 返回增强图和方法名。

如果要做“仅某种增强”的独立消融，应在 YAML 中把 `augmentation.methods` 设为只含一个
元素，而不是只把其他增强率设为 0。否则某些方法仍可能被选中并返回近似原图。

### 13.7 schema 合法不等于业务语义一定合理

当前约束保证增强后产生的关系三元组属于原图已观察集合，但没有保证：

- 新增边不是重复边；
- 不产生自环；
- 调用顺序仍满足时间逻辑；
- 节点类型交换符合真实实体身份；
- 新图对应某种真实故障。

`preserve_observed_schema=true` 只能防止明显越过类型模式，不能证明增强图是“合理异常”。
因此增强策略实验除了 AUC/AP，还应统计：

- 实际发生变化的图比例；
- 重复边和自环比例；
- schema 通过率；
- 每种增强在各数据集的有效率。

### 13.8 为什么增强图不参与 SVDD

增强图只进入 SSL 分支，没有被拉向 SVDD 正常中心。若把增强图也放入 SVDD：

```text
一边告诉模型“增强图是负样本”
一边又让增强图靠近正常中心
```

两个目标会直接冲突。当前实现用原图计算 SVDD，用原图与增强图共同计算 BCE，逻辑上更
一致。

## 14. 论文联合异常评分

### 14.1 从 Word 恢复的公式

```text
S_dist(G) = ||g - c||_2^2
S_ssl(G) = 1 - p
S_anomaly(G) = S_dist(G) * (1 + S_ssl(G))
```

恢复证据见 [评分公式恢复证据](评分公式恢复证据.md)。

### 14.2 代码

`model.py::HRAGNN.anomaly_score()`：

```python
svdd = (output.embedding - self.svdd_center).square().mean(dim=-1)
ssl_anomaly = 1.0 - output.ssl_logit.sigmoid()
return svdd * (1.0 + ssl_anomaly)
```

配置：

```yaml
evaluation:
  score_mode: paper_product
```

同时支持诊断模式：

```text
svdd
ssl
product
weighted_sum
paper_product
```

正式论文复现应使用 `paper_product`。

注意：`evaluation.score_ssl_weight` 只在 `weighted_sum` 模式下参与计算。当前主实验
使用 `paper_product`，所以即使 TraceLog 和 FlowGraph 的该配置值不同，也不会改变
论文乘法评分。真正控制 SSL 训练强度的是 `ssl.loss_weight`。

### 14.3 用数值例子理解最终分数

假设两张图：

```text
图 A：SVDD 距离 0.20，SSL 原图概率 0.90
图 B：SVDD 距离 0.15，SSL 原图概率 0.10
```

则：

```text
图 A：
S_ssl = 1 - 0.90 = 0.10
S = 0.20 * 1.10 = 0.22

图 B：
S_ssl = 1 - 0.10 = 0.90
S = 0.15 * 1.90 = 0.285
```

虽然 B 的 SVDD 距离更小，但它极不像原图，最终排序可以超过 A。

### 14.4 为什么分数越大越异常

两部分方向均已统一：

```text
SVDD 距离越大              -> 越异常
原图概率越小               -> 1-p 越大 -> 越异常
最终乘积越大               -> 越异常
label=1                    -> 指标中的正类异常
```

若接入新模型输出的是“正常分数越高”，必须像 SSL 一样先反向；否则 AUC 可能小于 0.5，
但那不一定说明模型毫无信息，也可能只是分数方向错误。

## 15. 断连图批处理

### 15.1 为什么需要

逐图前向时，双 RTX 4090 利用率只有约 20%，TraceLog 1000 图训练需要
115.5 秒。

### 15.2 如何合批

`graph.py::batch_graphs()`：

1. 拼接节点特征；
2. 按累计节点数平移每张图的边索引；
3. 拼接节点类型、边类型和边权；
4. 创建 `batch[node] = graph_index`；
5. 不添加任何跨图边。

### 15.3 如何保证不串图

- 消息传递只能沿 `edge_index`，不存在跨图边。
- 图级池化按 `batch` 分段。
- 关系注意力按目标节点分段，不按整个 batch 归一化。
- SSL 输出和 SVDD 分数保留每图维度。

测试 `test_batched_embeddings_match_individual_graphs()` 验证关闭 dropout 后，
逐图推理和合批推理的图表示数值一致。

### 15.4 性能结果

优化后 TraceLog 1000 图训练约 11.2 秒，约加速 10.3 倍；批量推理约从
20.7 ms/图降至 0.48 ms/图。

## 16. 训练流程

### 16.1 从命令到训练器

例如执行：

```bash
.venv/bin/python run.py train --config configs/tracelog.yaml
```

调用链：

```text
run.py
  -> cli.py::main()
  -> load_config(tracelog.yaml)
       -> 递归加载 base.yaml
       -> TraceLog 配置覆盖基础配置
  -> validate_config()
  -> Trainer(config)
  -> Trainer.train()
```

`Trainer.__init__()` 依次完成：

1. 固定 Python、NumPy、PyTorch 随机种子；
2. 选择 CUDA、MPS 或 CPU；
3. 加载完整数据集；
4. 加载官方划分；
5. 构造模型并移到设备；
6. 构造带独立随机状态的 augmentor；
7. 建立结果与 checkpoint 目录；
8. 保存解析后的完整 `config.yaml` 和 `splits.json`；
9. 创建 Adam 优化器和 BCE loss。

### 16.2 一个 epoch 的外层结构

```python
for epoch in range(1, maximum_epochs + 1):
    model.train()
    batches = _batches(train_indices, shuffle=True)
    for batch in batches:
        _train_batch(batch)

    if epoch % evaluate_every == 0:
        validation = evaluate("validation")
        更新 best checkpoint 和 early stopping

    写 history.csv
```

每个 epoch 都重新打乱训练索引。`max_train_graphs` 只用于 smoke test 或性能测试，正式
配置不应设置。

### 16.3 一个普通 batch 的 15 个步骤

#### 步骤 1：清空旧梯度

```python
self.optimizer.zero_grad(set_to_none=True)
```

`set_to_none=True` 减少内存写入；它不改变梯度数学含义。

#### 步骤 2：开启原型批次收集

```python
model.begin_prototype_batch()
```

模型的每一层建立空字典：

```text
layer._prototype_batch = {}
```

#### 步骤 3：合并原始图并移动到设备

```python
graph = batch_graphs([dataset[index] for index in indices]).to(device)
```

此处 `indices` 来自训练 split，官方数据中全部为正常图。

#### 步骤 4：原始图完整前向

```python
output = model(graph, update_prototypes=True)
```

内部顺序：

```text
类型投影
-> 关系 ID
-> Layer 1
-> Layer 2
-> max/mean/gate
-> graph_projection
-> ssl_head
```

输出：

```text
output.embedding         [B, D]
output.ssl_logit         [B]
output.node_embeddings   [N, H]
output.gate              [B, H]
```

#### 步骤 5：提交关系原型

```python
model.commit_prototype_batch()
```

每层、每关系把当前 batch 暂存值拼接后做一次 EMA。这个更新已经 `no_grad`。

#### 步骤 6：首次初始化 SVDD 中心

只在整个训练第一次：

```python
if not model.svdd_center_initialized:
    model.initialize_svdd_center(embeddings)
```

后续 epoch 和 batch 都不再更新中心。

#### 步骤 7：计算 SVDD 损失

```python
svdd_loss = (embeddings - center).square().mean()
```

对 batch 维和输出维一起求均值，得到标量。

#### 步骤 8：在 CPU 为每张原图构造增强图

```python
augmented_graph_i = augmentor.augment(dataset[index]).graph
```

每张图随机选择一种允许的增强。增强是离散操作，不在 autograd 图中。

#### 步骤 9：合并增强图并移到设备

```python
augmented = batch_graphs(augmented_graphs).to(device)
```

增强图和原图的图数量相同，但节点/边数可能因边添加发生变化。

#### 步骤 10：增强图完整前向

```python
augmented_output = model(augmented, update_prototypes=False)
```

模型参数仍参与梯度，只是禁止增强图更新 prototype。

#### 步骤 11：构造 SSL 伪标签

```text
original_logits   [B] -> labels 全 1
augmented_logits  [B] -> labels 全 0
```

拼接为 `[2B]`。

#### 步骤 12：计算 BCE

```python
ssl_loss = BCEWithLogitsLoss(logits, labels)
```

`BCEWithLogitsLoss` 内部已包含 sigmoid，训练时不能先手工 sigmoid。

#### 步骤 13：组合损失

```python
total_loss = svdd_loss + ssl_loss_weight * ssl_loss
```

TraceLog 的 `ssl_loss_weight=0.001`，FlowGraph 为 `0.21`。二者差异很大，说明论文结果
对数据集专属超参数敏感。

#### 步骤 14：反向传播和梯度裁剪

```python
total_loss.backward()
clip_grad_norm_(model.parameters(), 5.0)
```

裁剪的是全模型参数梯度总范数，原型和中心不是 parameter，因此不参与。

#### 步骤 15：Adam 更新

```python
optimizer.step()
```

更新类型投影、关系矩阵、注意力 MLP、root、LayerNorm、门控、图投影和 SSL head。

### 16.4 训练伪代码

```python
for epoch in epochs:
    shuffle(train_graph_indices)

    for indices in batches:
        optimizer.zero_grad()

        # A. 原始正常图
        model.begin_prototype_batch()
        original = batch_graphs(dataset[i] for i in indices)
        out = model(original, update_prototypes=True)
        model.commit_prototype_batch()

        if center_not_initialized:
            center = detach(mean(out.embedding))

        loss_svdd = mean((out.embedding - center) ** 2)

        # B. 人工增强图
        augmented = batch_graphs(augment(dataset[i]) for i in indices)
        aug_out = model(augmented, update_prototypes=False)

        logits = concat(out.ssl_logit, aug_out.ssl_logit)
        targets = concat(ones(B), zeros(B))
        loss_ssl = BCEWithLogits(logits, targets)

        # C. 共享参数联合优化
        loss = loss_svdd + eta * loss_ssl
        loss.backward()
        clip_gradients()
        optimizer.step()
```

### 16.5 参数究竟由哪项损失训练

| 模块 | SVDD 梯度 | SSL 梯度 | EMA/手工更新 |
|---|---:|---:|---:|
| 类型投影 | 是 | 是 | 否 |
| 关系消息矩阵 | 是 | 是 | 否 |
| 语义 attention MLP | 是 | 是 | 否 |
| 门控读出 | 是 | 是 | 否 |
| 图投影 | 是 | 是 | 否 |
| SSL head | 否 | 是 | 否 |
| 关系 prototype/scale | 否 | 否 | 是 |
| SVDD center | 否 | 否 | 首批初始化一次 |

SSL head 不在 SVDD 路径上，所以只由 BCE 训练；编码器和读出是两项损失共享的。

### 16.6 优化器配置

```text
Adam
learning_rate：TraceLog 0.0001，FlowGraph 0.01
weight_decay：1e-6
gradient_clip：5.0
```

FlowGraph 学习率是 TraceLog 的 100 倍，也是随机种子不稳定性需要检查的因素之一。

## 17. 验证、早停和测试

每个 epoch 后执行验证：

- 有验证集时监控验证 AUC。
- 没有验证集时监控负训练损失。
- 连续 `early_stopping_patience` 次未改善则停止。
- 保存 `best.pt`。
- 训练结束加载 `best.pt` 后测试。

测试阶段分别保存：

```text
score
svdd_score
ssl_anomaly_score
ssl_probability
gate_mean
```

关系诊断默认关闭，因为逐图提取关系统计昂贵。开启：

```bash
--set evaluation.collect_diagnostics=true
```

开启后保存每张图、每种关系的：

```text
mean_attention
mean_deviation
prototype_updates
```

### 17.1 `train()`、`validation`、`test` 三种模式

| 模式 | `model.training` | Dropout | 更新原型 | 反向传播 |
|---|---:|---:|---:|---:|
| 训练原图 | True | 开启 | 是 | 是 |
| 训练增强图 | True | 开启 | 否 | 是 |
| 验证/测试 | False | 关闭 | 否 | 否 |

`evaluate()` 由 `@torch.no_grad()` 装饰，因此验证和测试不会建立梯度图。

### 17.2 TraceLog 如何选择 best checkpoint

每个 epoch：

```text
计算 validation score
-> 用图级真值计算 validation AUC
-> 如果 AUC 提升，保存 best.pt
-> 连续 patience 次不提升则停止
```

这使用了验证异常标签进行模型选择。它没有把标签用于参数梯度，但属于带标签验证，论文应
明确说明，不能写成完全无标签模型选择。

### 17.3 FlowGraph 如何选择 best checkpoint

FlowGraph 没有 validation 文件。`evaluate("validation")` 返回 NaN，训练器改用：

```text
monitored = -training_loss
```

即训练 loss 越低越好。这不使用异常标签，但存在另一个风险：训练目标下降不保证测试 AUC
提升。首批次中心错误时，模型可能很好地优化一个错误目标。

### 17.4 测试阶段的详细流程

训练停止后：

1. 加载 `best.pt`；
2. 按 evaluation batch size 合批测试图；
3. 前向计算 embedding、SSL logit 和 gate；
4. 计算 SVDD、SSL 和联合异常分数；
5. 收集图级真值；
6. 计算 AUC/AP；
7. 写 `test_predictions.csv`；
8. 可选逐图再次前向并写 `test_relations.csv`；
9. 写 `metrics.json`。

关系诊断要逐图再次前向，是因为批量 `relation_diagnostics` 当前按整个断连 batch 汇总同一
关系，无法直接拆回每张图。为保证每图解释正确，诊断模式牺牲速度。

### 17.5 AUC 与 AP 分别回答什么

```text
AUC：
随机抽一张异常图和一张正常图，
异常分数排在正常图前面的概率。

AP：
按异常分数从高到低排序后，
对异常类 precision-recall 曲线的汇总。
```

在异常比例较低时，AP 比 AUC 更敏感。两者只依赖排序，不要求分数落在 `[0,1]`。

### 17.6 checkpoint 的一个实现细节

`best.pt` 在验证指标改善时保存。训练结束后代码先加载 `best.pt`，再保存 `last.pt`。
因此当前 `last.pt` 的**模型权重实际上也是 best 权重**，但 optimizer state 仍是训练结束时
状态。它不是严格意义上的“最后 epoch 可续训 checkpoint”。

目前代码也没有恢复 optimizer 后继续训练的入口，所以文档和论文应主要引用 `best.pt`，
不要把 `last.pt` 描述为完整断点续训状态。

## 18. 实验结果与绘图分离

训练输出：

```text
config.yaml
splits.json
history.csv
metrics.json
test_predictions.csv
checkpoints/best.pt
checkpoints/last.pt
```

可选输出：

```text
test_relations.csv
```

绘图代码只读取 CSV，不调用模型训练。这样可以：

- 重画图而不重复训练；
- 保留原始数值；
- 修改配色不污染实验；
- 审查图中的每个柱子来自哪个 CSV。

五种子汇总使用 pandas `groupby` 计算均值和样本标准差。

### 18.1 每个结果文件回答什么

| 文件 | 内容 | 审查用途 |
|---|---|---|
| `config.yaml` | 完整解析后的实际参数 | 防止只看父 YAML |
| `splits.json` | 本次运行的图索引 | 核对所有方法是否同划分 |
| `history.csv` | 每 epoch loss、验证指标、耗时 | 判断早停和收敛 |
| `metrics.json` | 最终指标、参数量、耗时、设备 | 套件恢复和汇总 |
| `test_predictions.csv` | 每张图的真值和各分数组件 | 重新算指标、画分布图 |
| `test_relations.csv` | 每图每关系的注意力与偏差 | 解释性分析 |
| `best.pt` | 最佳模型、配置、optimizer state | 最终评估 |

### 18.2 `test_predictions.csv` 的列

```text
graph_id             原始图 ID
label                0 正常、1 异常
score                最终联合异常分数
svdd_score           图表示到中心的距离
ssl_anomaly_score    1 - sigmoid(ssl_logit)
ssl_probability      sigmoid(ssl_logit)，越大越像原图
gate_mean            逐维 gate 的均值，仅用于粗略观察
```

`gate_mean` 会丢掉 128 或 32 个维度的细节，不能只凭一个均值证明模型选择了 max 或 mean。
要做严谨分析，应导出完整门向量或按维度统计。

### 18.3 为什么图片不能作为原始证据

PNG 只保留像素，不保留精确数值、seed 和配置。正确证据链是：

```text
config.yaml + splits.json
        |
        v
checkpoint + test_predictions.csv
        |
        v
metrics.json / suite runs.csv
        |
        v
summary.csv
        |
        v
plotting.py -> PNG
```

任何图中数字都应能沿这条链反查，而不是根据图片手工录回论文表格。

## 19. 基线实现及其边界

### 19.1 OCHetGCN 和 HRGCN

通过 HRA-GNN 的配置化退化实现：

- 静态关系融合；
- 最大池化；
- 按对应配置启用或关闭 SSL。

其中静态拼接使用分块线性等价实现，不显式构造巨大拼接张量。

### 19.2 HGT

实现类型关系矩阵调制的 query/key/value 注意力，最大池化后使用 DeepSVDD。

这是 HGT 风格适配器，不保证覆盖官方 HGT 的多头注意力、相对时间编码或全部
训练细节。

### 19.3 DeepTraLog

实现：

- 拓扑消息聚合；
- GRUCell 更新；
- 节点注意力读出；
- DeepSVDD 图级异常评分。

这是面向公共协议的近似实现，不是 DeepTraLog 作者代码。

### 19.4 GLocalKD

实现固定随机教师和可训练学生：

```text
global_loss = 教师/学生图表示差异
local_loss  = 教师/学生节点表示差异
score       = global_loss + local_loss
```

同样属于适配实现。

### 19.5 结果标注要求

这三种适配器的结果必须写成：

```text
统一协议下的适配器复现结果
```

不能写成：

```text
官方源码复现结果
```

要做严格 SOTA 对比，必须进一步运行各方法官方仓库，并统一数据划分、输入特征和
异常比例。

## 20. 配置与复现实验

### 20.1 配置继承

`config.py::load_config()` 支持：

```yaml
extends: base.yaml
```

数据集 YAML 只覆盖不同参数。命令行 `--set key=value` 可继续覆盖。

每次训练都会把解析后的完整配置写到运行目录的 `config.yaml`，避免以后不知道
当时到底用了什么参数。

### 20.2 五随机种子

实验套件定义：

```yaml
seeds: [11, 22, 33, 44, 55]
```

每个 seed 有独立：

- 参数初始化；
- 数据顺序；
- 增强随机序列；
- SVDD 首批次中心；
- checkpoint 和预测文件。

### 20.3 断点续跑

如果目标运行目录已有 `metrics.json`，实验套件默认读取已有结果而不重训。

该机制只判断任务是否完成，不支持从某个 epoch 恢复优化器继续训练。中断于
epoch 中间的任务仍需从该 seed 开始重训。

### 20.4 单次实验从零到结果

以 FlowGraph seed 42 为例：

```bash
.venv/bin/python run.py train \
  --config configs/flowgraph.yaml \
  --set training.seed=42 \
  --set output.run_name=hra_full
```

解析后的输出目录：

```text
artifacts/results/
  FlowGraph/
    hra_full/
      seed_42/
        config.yaml
        splits.json
        history.csv
        metrics.json
        test_predictions.csv
        checkpoints/
          best.pt
          last.pt
```

目录由：

```text
dataset.name / output.run_name / seed
```

共同决定。如果只改参数但不改 `run_name`，可能覆盖或错误复用旧实验。

### 20.5 五随机种子套件如何展开

`configs/experiments/full_flowgraph.yaml`：

```yaml
name: full_flowgraph
base_config: ../flowgraph.yaml
seeds: [11, 22, 33, 44, 55]
variants:
  HRA-GNN: {}
```

`run_experiment_suite()` 执行的是笛卡尔积：

```text
每个 variant
  × 每个 seed
```

对每个任务：

1. 读取 base config；
2. 合并 variant 覆盖项；
3. 覆盖 `training.seed`；
4. 设置独立 `run_name`；
5. 训练或读取已有 `metrics.json`；
6. 把最终指标加入 `runs.csv`；
7. 按 `(dataset, variant)` 分组计算 mean/std。

运行命令：

```bash
.venv/bin/python run.py experiment \
  --suite configs/experiments/full_flowgraph.yaml

.venv/bin/python run.py experiment \
  --suite configs/experiments/full_tracelog.yaml
```

### 20.6 为什么要五个 seed

随机种子同时控制：

- 参数初始化；
- 每个 epoch 的训练图顺序；
- 首批次 SVDD 中心；
- 每张图选择哪一种增强；
- 增强修改哪些节点或边；
- Dropout mask。

因此 seed 不只是“让结果有一点噪声”。在当前模型中，它会改变正常中心和整个优化轨迹。
FlowGraph 的五次 AUC 跨越 `0.0308` 到 `0.9837`，单次结果无法代表模型能力。

### 20.7 现有五种子主实验是怎样跑的

远程双 RTX 4090 环境中，将不同 seed 配置分配到两张卡。代码层面每个进程仍是单卡训练，
双卡用于并行运行不同实验，不是一次训练做分布式数据并行。

合理理解：

```text
GPU 0：seed 11、33、55 等独立进程
GPU 1：seed 22、44 等独立进程
```

每个进程：

- 有独立随机种子；
- 有独立结果目录；
- 不共享模型参数；
- 最后只在 suite CSV 中汇总。

这与“把一个 batch 拆到两张卡共同训练同一个模型”不同。

### 20.9 `metrics.json` 恢复机制的风险

套件发现目标目录已有 `metrics.json` 时默认直接读取，不验证：

- 当前代码 commit 是否改变；
- base YAML 是否改变；
- 命令行参数是否改变；
- checkpoint 是否仍存在。

因此修改模型后重跑必须：

```text
方法 1：使用新的 output.run_name
方法 2：明确使用 experiment --force
```

优先推荐新 `run_name`，因为它保留旧结果，便于审计。`--force` 会在相同目录重新运行，
可能覆盖历史证据。

- 前向、反向和保存流程能结束；
- GPU 不 OOM；
- loss 不是 NaN；
- checkpoint、history、prediction 和 metrics 文件能生成。

它的 AUC/AP 不具备论文意义。

#### 阶段 4：单 seed 全量运行

先运行一个完整 seed，检查训练时长、显存、早停行为、分数方向和预测文件。单 seed
失败时立即排查，不应直接复制成五次失败。

#### 阶段 5：五 seed 并行

两张 GPU 各启动独立进程，每个进程通过 `CUDA_VISIBLE_DEVICES` 只看到一张卡。任务
按预计耗时分配，避免两进程同时写同一 `run_name/seed` 目录。

监控项包括：

```text
nvidia-smi：显存、GPU 利用率、异常退出
history.csv：loss、验证 AUC、epoch 时间
进程返回码：是否正常结束
metrics.json：是否写出最终指标
```

当前实测平均训练时间：

```text
TraceLog：4925.77 秒/seed，约 82.1 分钟
FlowGraph：505.08 秒/seed，约 8.4 分钟
```

这是五次运行的平均值，不是理论复杂度，也不能直接外推到其他 GPU。

#### 阶段 6：汇总和异常检查

五次运行结束后检查：

1. `runs.csv` 是否恰好包含预期的 5 个 seed；
2. 是否有 NaN、重复 seed 或读取旧 `metrics.json`；
3. mean/std 是否由全部 seed 计算；
4. 联合、SVDD、SSL 三种分数是否方向一致；
5. 极端 seed 是否应保留。

seed 55 这类低结果必须保留。排除某个 seed 只有在进程错误、文件损坏或明确违反实验
协议时才成立，不能因为指标低而删除。

#### 阶段 7：最后绘图

确认 CSV 后才运行绘图命令。图是结果的展示层，不参与训练、选 seed 或重新计算模型
输出。修改配色、字体和尺寸不应改变 CSV。

### 20.13 一次实验如何形成可审计证据链

以一个 seed 为例，证据顺序是：

```text
YAML + 命令行覆盖
  -> 运行目录中的 config.yaml
  -> splits.json
  -> history.csv
  -> best.pt
  -> test_predictions.csv
  -> metrics.json
  -> suite runs.csv
  -> variant_summary.csv
  -> 论文表格或图片
```

审查时应反向追踪。例如论文中的 TraceLog AUC 均值 `0.8113`：

1. 在 `variant_summary.csv` 找到未四舍五入均值；
2. 在 `runs.csv` 找到五个 seed 的 AUC；
3. 在各 seed 的 `metrics.json` 核对单次结果；
4. 必要时从 `test_predictions.csv` 的 `label` 和 `score` 重算；
5. 用该运行目录的 `config.yaml` 和 `splits.json` 确认协议。

这样才能区分“绘图抄错”“汇总错误”“评分错误”和“模型本身表现不佳”。

## 24. 训练过程中究竟“学”了什么

理解训练最容易出错的地方，是把所有状态都当成普通神经网络参数。实际上，当前
实现同时维护四类状态。

### 24.1 四类状态及其更新方式

| 状态 | 例子 | 如何更新 | 是否由反向传播更新 |
|---|---|---|---|
| 可训练参数 | 类型投影、关系矩阵、注意力向量、门控层、SSL 头 | Adam | 是 |
| 关系统计状态 | 每层每种关系的均值、标准差、初始化标记 | 原图 batch 的 EMA | 否 |
| SVDD 中心 | `svdd_center` | 第一个训练 batch 计算一次后固定 | 否 |
| 优化过程状态 | Adam 一阶/二阶矩、epoch、best metric | 优化器与训练循环 | 否 |

这四类状态回答不同问题：

- 可训练参数回答“怎样把输入图映射到有用表示”；
- 关系原型回答“正常训练图中，每种关系通常产生什么消息”；
- SVDD 中心回答“当前表示空间中，正常图集中在哪里”；
- 优化器状态回答“参数下一步该往哪里走、走多远”。

为了重现测试行为，checkpoint 的 `model_state` 必须包含普通参数、关系原型和 SVDD
中心；当前实现通过 PyTorch buffer 把后两者一并保存。为了从中断处严格继续训练，
还需要恢复 optimizer 状态、epoch 和 early-stopping 状态。当前 checkpoint 虽保存
optimizer 状态，但 `load_checkpoint()` 只加载 `model_state`，所以它支持评估，不是
严格的训练续跑。

### 24.2 训练开始前

开始第一个 epoch 前：

1. 类型投影、关系矩阵、注意力和门控等参数随机初始化；
2. 每层关系原型均值和标准差尚未初始化；
3. SVDD 中心不存在；
4. Adam 尚无历史梯度；
5. 数据划分已固定，训练索引只指向正常图；
6. 每个 epoch 内打乱训练索引，但不改变 train/val/test 归属。

此时模型还不知道什么是“正常”。“正常性”不是人工写入规则，而是由正常训练图
共同塑造的表示、关系原型和 SVDD 中心。

### 24.3 第一个训练 batch

第一批是特殊的：

1. 原型未初始化，因此原图前向时所有关系偏差按 0 处理；
2. 关系注意力暂时只依赖基础语义分数；
3. 前向结束后，用该批原图初始化出现过的关系原型；
4. 用该批图 embedding 的均值初始化 SVDD 中心；
5. 中心立即 `detach`，因此不会随梯度移动；
6. 仍然计算该批 SVDD 损失、SSL 损失并更新网络参数。

第一批因此同时决定了两个重要坐标系的起点：关系正常原型和图级 SVDD 中心。
如果第一批样本规模小、结构极端或顺序不稳定，后续结果可能明显变化。

### 24.4 第二个及后续训练 batch

从第二批开始：

1. 原图关系消息会与已有正常原型比较；
2. 偏离程度进入关系注意力 logits；
3. 原图前向结束后，当前批统计通过 EMA 更新原型；
4. SVDD 中心保持不变；
5. 参数继续由 `SVDD + lambda_ssl * SSL` 更新；
6. 下一个 batch 看到的是更新后的网络参数和关系原型。

需要特别区分：

```text
网络参数：每 batch 由梯度更新
关系原型：每 batch 由统计量更新
SVDD 中心：第一 batch 后固定
```

### 24.5 一个 epoch 结束后

一个 epoch 的训练 batch 全部完成后：

1. 网络已执行多次 Adam 更新；
2. 关系原型已吸收该 epoch 所有原图 batch；
3. TraceLog 在验证集计算有标签 AUC/AP；
4. FlowGraph 因无验证集，使用训练损失作为监控量；
5. 如果监控指标改善，则保存 best checkpoint；
6. 如果长期无改善，则触发 early stopping。

验证和测试不反向传播，也不更新关系原型。否则评估数据会改变模型状态，后面的图
会受到前面的评估图影响。

## 25. 从一张图到一个异常分数：完整跟踪

下面不按文件，而按数据实际流动顺序解释。设 batch 中有两张图：

```text
图 A：N_A 个节点、E_A 条边
图 B：N_B 个节点、E_B 条边
隐藏维：H
关系总数：R = T_v * T_e * T_v
```

### 25.1 原始数据阶段

每张图至少包含：

```text
x           [N, F]   节点特征
node_type   [N]      节点类型编号
edge_index  [2, E]   源节点和目标节点编号
edge_type   [E]      边类型编号
label       标量     图级真实标签，仅评估使用
graph_id    标识     用于追踪原始样本
```

`label=0` 表示正常图，`label=1` 表示异常图。代码没有节点级异常标签，也没有告诉
模型“第几条边异常”。因此，模型只能输出整张图的异常分数；关系注意力可用于分析，
但不是经过关系级真值监督得到的定位器。

### 25.2 合批阶段

`batch_graphs` 把 A、B 的节点沿第 0 维拼接：

```text
x_batch.shape = [N_A + N_B, F]
```

B 的边端点统一加 `N_A`，并创建：

```text
batch = [0, 0, ..., 0, 1, 1, ..., 1]
```

这里的 `batch[i]` 表示节点 `i` 属于哪张图。A、B 之间没有新增边，所以消息传递
不会跨图；图级读出则按 `batch` 分段聚合。

### 25.3 节点类型投影

原始特征维度为 `F`，模型隐藏维度为 `H`。对类型 `t` 的节点使用独立线性层：

```text
h_i^(0) = W_type[t_i] x_i + b_type[t_i]
```

输出：

```text
h^(0).shape = [N_A + N_B, H]
```

这样做是因为不同类型节点的同一原始特征列未必具有相同语义。类型专属投影先把
它们送入统一隐藏空间。

### 25.4 第 1 个关系偏差层

对于每条边 `e=(u,v)`：

1. 根据源节点类型、边类型、目标节点类型计算关系 `r_e`；
2. 用关系专属矩阵变换源节点表示；
3. 对相同目标节点、相同关系的消息取均值；
4. 与该层对应关系的正常原型计算标准化偏差；
5. 用基础语义分数加偏差调制项形成 attention logit；
6. 在同一目标节点拥有的关系通道之间做 softmax；
7. 加权求和关系消息；
8. 加上目标节点自身的 root 分支；
9. 经过残差、激活、dropout 和归一化得到 `h^(1)`。

输出仍是：

```text
h^(1).shape = [N_A + N_B, H]
```

节点数没有改变，改变的是每个节点的表示。第 1 层让节点吸收一跳邻居信息。

### 25.5 第 2 个关系偏差层

第 2 层重复同样流程，但输入改为 `h^(1)`，并使用第 2 层自己维护的关系矩阵和
关系原型。输出：

```text
h^(2).shape = [N_A + N_B, H]
```

两层后，一个节点一般能融合两跳邻域信息。第 1 层和第 2 层的“正常关系消息”
分布不同，不能共享同一套原型。

### 25.6 图级读出

按 `batch` 分别对图 A、B 的节点表示计算：

```text
mean_pool = 每图节点均值       [2, H]
max_pool  = 每图节点逐维最大值 [2, H]
gate      = sigmoid(MLP(...))  [2, H]
z         = gate * max_pool + (1 - gate) * mean_pool
```

于是：

```text
z.shape = [2, H]
```

`mean_pool` 表示整体状态，`max_pool` 强调少数强响应节点。逐维门控允许某些隐藏
维更依赖局部峰值，另一些维更依赖全局平均。

### 25.7 两个训练分支

原图 embedding `z_original` 进入两处：

```text
z_original -> 与 SVDD 中心计算距离
z_original -> SSL 分类头，目标为 1
```

每张原图随机增强后重新完整前向：

```text
augmented graph -> encoder -> z_augmented -> SSL 分类头，目标为 0
```

注意，增强不是在 embedding 上加噪声，而是修改图结构或类型后，再从节点投影开始
完整运行所有 GNN 层。

### 25.8 反向传播

总损失：

```text
L = L_svdd + lambda_ssl * L_ssl
```

反向传播时：

- SVDD 梯度只从原图路径回到 encoder；
- SSL 原图梯度从分类头回到 encoder；
- SSL 增强图梯度也从分类头回到同一个 encoder；
- 两次 encoder 前向共享同一组参数，梯度在参数上相加；
- 关系原型和 SVDD 中心不接收梯度；
- 图真实标签不参与损失。

### 25.9 测试阶段

测试不构造增强图，只对原图前向。对每张图计算：

```text
S_dist = mean((z - center)^2)
S_ssl  = 1 - sigmoid(ssl_head(z))
S_final = S_dist * (1 + S_ssl)
```

最后才读取测试图真实标签，用全部 `S_final` 与 `label` 计算 AUC/AP。也就是说：

```text
模型产生分数时不看标签；
评价分数好坏时才看标签。
```

## 26. 核心创新的“为什么、实现、证据、风险”总表

| 模块 | 为什么需要 | 当前如何实现 | 应提供什么证据 | 主要风险 |
|---|---|---|---|---|
| 异构关系三元组 | 同一边类型在不同端点类型间语义可能不同 | `(src_type, edge_type, dst_type)` 编码 | 与只用 edge type 对比 | 关系数平方增长 |
| 关系专属消息变换 | 不同关系应提取不同模式 | 每层每关系一个 `H x H` 矩阵 | 参数量和性能消融 | TraceLog 参数量很大 |
| 关系正常原型 | 给“关系是否反常”建立正常参照 | 正常原图统计的 EMA 均值/标准差 | momentum、初始化、污染实验 | 多峰分布被均值压平 |
| 标准化关系偏差 | 不同关系尺度不可直接比较 | 各维 z-score 的均方根 | 原始距离与标准化距离对比 | 丢失偏离方向 |
| 偏差调制注意力 | 让异常关系在融合时获得不同权重 | 语义 logit 加 `gamma * deviation` | 权重可视化、gamma 消融 | 高偏差不一定等于异常 |
| 门控混合读出 | 同时覆盖局部峰值和全局改变 | 逐图逐维混合 mean/max | mean、max、固定混合消融 | max 对噪声敏感 |
| 自监督增强 | 在无异常训练样本时增加判别信号 | 原图/增强图二分类 | 四种增强独立消融 | 可能只学习增强伪影 |
| SVDD | 只用正常图建立图级正常区域 | 首批中心固定、最小化均方距离 | 多中心初始化和方差分析 | 首批依赖严重 |
| 乘法联合评分 | 用 SSL 置信度调整距离分数 | `dist * (1 + ssl_anomaly)` | 加法/乘法/仅 SVDD 对比 | SSL 最多只放大 2 倍 |

其中，前三个关系模块加上偏差调制注意力，才构成“关系偏差”主线；门控读出解决
图级局部/全局汇总；SVDD 和 SSL 负责训练与打分。不能把所有模块笼统称为一个
创新，也不能仅凭整模结果证明每个模块都有效。

## 28. 术语表与常见误解

| 术语 | 在本项目中的准确含义 |
|---|---|
| 图 | 一条 TraceLog trace 或一个 FlowGraph flow 样本形成的异构图 |
| 节点 | 图中的事件、实体或数据记录，具体语义由数据集决定 |
| 边 | 两个节点之间的有向连接 |
| 节点类型 | 节点所属类别的离散编号 |
| 边类型 | 数据中显式提供的边类别；FlowGraph 当前发布文件没有该列 |
| 关系 | 源节点类型、边类型、目标节点类型组成的三元组 |
| 关系通道 | 某个目标节点收到的某一种关系的聚合消息 |
| layer | 一次关系消息传递和节点更新；不是一个完整模型 |
| embedding | 图经全部 GNN 层和读出后得到的定长向量 |
| 原型 | 正常训练图中某层某关系消息的运行统计，不是训练样本 |
| SVDD 中心 | 第一个训练 batch 图 embedding 的固定均值 |
| 真实标签 | 数据集提供的图级正常/异常标记，只用于划分或评估 |
| SSL 伪标签 | 人工规定的原图 1、增强图 0，不代表真实正常/异常 |
| logit | sigmoid 之前可为任意实数的分类头输出 |
| anomaly score | 测试时用于排序样本异常程度的连续数值 |
| batch | 一次共同前向和反向传播的多张断连图 |
| epoch | 训练划分中的图被完整遍历一次 |
| seed | 控制初始化、打乱和增强随机性的随机种子 |
| dataset index | 图在加载后列表中的位置，用于划分 |
| graph ID | 原始数据中的样本标识，用于追踪，不一定等于 dataset index |

最后用六个判断句固定整体认识：

1. **这是图级异常检测，不是节点级或边级异常定位。**
2. **训练集只含正常图，异常图标签不进入训练损失。**
3. **一张图先经过两层关系消息传递，再聚合为一个图 embedding。**
4. **关系原型由统计更新，网络参数由梯度更新，两者不是同一种学习。**
5. **训练每个 batch 有原图和增强图两次前向，测试只有原图一次前向。**
6. **AUC/AP 是最终评估指标，不是神经网络直接优化的损失。**

## 29. 高基数节点类型为什么需要 edge-only 关系

原论文数据的 canonical 关系定义为：

```text
relation_id = (source_type * num_edge_types + edge_type)
              * num_node_types + target_type
```

关系总数是：

```text
num_node_types * num_edge_types * num_node_types
```

TraceLog 的 8 种节点类型和 4 种边类型只产生 256 种关系，适合这种设计。
ADFA-LD 的训练词表有 151 种 syscall，若照搬则产生：

```text
151 * 3 * 151 = 68,403
```

每种关系还需要 `hidden_dim x hidden_dim` 消息矩阵。两层 128 维模型在参数转入
GPU 时就会超过 V100 16GB，调小 batch 无法解决，因为 OOM 发生在读取训练图之前。

ADFA-LD 配置使用：

```yaml
model:
  relation_schema: edge_only
```

此时：

- `type_projection[syscall_id]` 仍保留 syscall 的类别差异；
- 关系消息矩阵只按 3 种边语义共享；
- 原型也从 68,403 组减少到 3 组；
- 参数复杂度由 `O(T^2 R H^2)` 降为 `O(R H^2)`。

这个修改不是把所有节点变成同一种节点。节点输入投影仍按 syscall 区分，只是避免
为每一对 syscall 再单独学习一张消息矩阵。TraceLog、FlowGraph 和 HDFS 默认继续
使用 `canonical`，因此该适配不会改变原论文实验行为。
