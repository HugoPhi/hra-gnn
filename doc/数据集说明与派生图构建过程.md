# 数据集说明与派生图构建过程

更新时间：2026-07-02

## 1. 文档用途与归属边界

本文档用于撰写论文中的“数据集”“数据预处理”和“图构建”章节。

必须区分以下三类工作：

1. **原始数据采集：**TraceLog、FlowGraph、HDFS 和 ADFA-LD 均不是本项目采集；
2. **已有图数据复用：**TraceLog 和 FlowGraph 直接使用 HRGCN 作者发布的已处理图数据；
3. **派生图数据制作：**本项目将公开 HDFS 日志和 ADFA-LD 系统调用轨迹转换为图数据。

因此，论文中不应写“本文采集/创建了 HDFS 或 ADFA-LD 数据集”，建议写：

> 本文基于公开 HDFS 和 ADFA-LD 数据构建了面向图级异常检测的派生图数据表示。

派生图数据只改变数据表示，不改变原始样本来源和真值标签的归属。

## 2. 统一任务定义

四个数据集均被统一为单类图级异常检测任务。每个样本是一张有向属性图：

```text
G = (V, E, X, T_v, T_e, y)
```

其中：

- `V`：节点集合；
- `E`：有向边集合；
- `X in R^(N x F)`：节点特征矩阵；
- `T_v in N^N`：节点类型编号；
- `T_e in N^E`：边类型编号；
- `y in {0,1}`：图标签，0 表示正常，1 表示异常。

训练集只包含正常图。异常标签不进入训练损失，只在最终评估时用于计算
AUROC、AP、F1、MCC 等指标。

## 3. 数据集总体统计

| 数据集 | 数据范围 | 图数 | 正常图 | 异常图 | 特征维 | 节点类型 | 显式边类型 | 平均节点数 | 平均边数 |
|---|---|---:|---:|---:|---:|---:|---:|---:|---:|
| TraceLog | HRGCN 发布数据 | 132485 | 109151 | 23334 | 7 | 8 | 4 | 204.633 | 224.008 |
| FlowGraph | HRGCN 发布数据 | 600 | 500 | 100 | 26 | 8 | **1（发布文件实际值）** | 8411.463 | 12730.403 |
| HDFS-100k | 公开结构化子集的派生图 | 7940 | 7627 | 313 | 8 | 19 | 3 | 13.201 | 30.265 |
| ADFA-LD | 完整公开轨迹的派生图 | 5951 | 5205 | 746 | 8 | 151 | 3（其中一类当前冗余） | 461.696 | 1363.159 |

注意：

- HDFS 当前是 10 万行结构化日志筛查子集，不是完整 HDFS_v1；
- HRGCN 论文称 FlowGraph 有 26 种边类型，但实际发布的 `edge_index.csv`
  没有 `edge_type` 列；26 维数据位于节点特征文件中；
- 因此当前可执行 FlowGraph 表示只有一种显式边类型，不能在论文中继续写成
  “使用发布数据中的 26 种边类型”。

## 4. 统一张量格式

模型实际读取的单张图包含：

```text
graph_id       标量          样本追踪编号
x              [N, F]        节点特征
edge_index     [2, E]        有向边的源节点和目标节点编号
node_type      [N]           节点类型
edge_type      [E]           边类型
edge_weight    [E] 或 None   边权
label          标量          0 正常，1 异常
```

`edge_index` 的第一行是源节点编号，第二行是目标节点编号：

```text
edge_index = [[src_1, src_2, ..., src_E],
              [dst_1, dst_2, ..., dst_E]]
```

它使用 `[2,E]` 而不是 `[E,2]`，是为了兼容 PyTorch Geometric 的稀疏消息传递接口。

## 5. TraceLog

### 5.1 原始背景

TraceLog 是由列车票务微服务系统运行轨迹派生的大规模异构图数据集。HRGCN
论文说明该数据包含 4 个故障大类和 14 个故障子类，例如异步交互、多实例、
配置相关和单体化故障。14 类故障轨迹统一作为异常图。

本项目没有获得原始微服务日志到图的完整生成程序，直接使用 HRGCN 仓库发布的
预处理图文件。因此，本文只能说明发布图文件如何被加载，不能声称独立复现了
TraceLog 的原始构图过程。

### 5.2 已发布图文件

| 文件 | 含义 |
|---|---|
| `node_feature_norm.csv` | 每个节点的 7 维归一化属性 |
| `edge_index.csv` | 源节点、目标节点、边类型、图 ID 和权重 |
| `node_types.txt` | 每张图中 8 类节点分别包含哪些节点编号 |
| `trace_info.csv` | 图 ID、正常/异常标记和故障类别 |
| `model_gid_list_train.txt` | 训练图 ID |
| `model_gid_list_eval.txt` | 验证图 ID |
| `model_gid_list_test.txt` | 测试图 ID |

7 个节点属性在发布文件中只命名为 `node_attr_0` 至 `node_attr_6`，没有提供
可核验的业务字段名称。论文中应称其为“发布数据提供的 7 维归一化节点属性”，
不要自行解释为延迟、CPU、内存等具体指标。

### 5.3 异构性

TraceLog 同时具有：

- 8 种节点类型；
- 4 种显式边类型；
- 关系三元组 `(源节点类型, 边类型, 目标节点类型)`。

canonical 关系空间最多为：

```text
8 * 4 * 8 = 256
```

不同类型节点的原始特征列数相同，不代表图是同构图。异构性由离散
`node_type` 和 `edge_type` 决定，统一特征维度只是为了支持批量矩阵计算。

### 5.4 标签与划分

`trace_bool=true` 表示正常，转换为标签 0；其他值转换为异常标签 1。

| 划分 | 图数 | 正常 | 异常 |
|---|---:|---:|---:|
| 训练 | 65000 | 65000 | 0 |
| 验证 | 33742 | 22075 | 11667 |
| 测试 | 33743 | 22076 | 11667 |

当前复现严格读取发布的图 ID 划分，不重新随机划分。

## 6. FlowGraph

### 6.1 原始背景

FlowGraph 是系统流行为图数据集。HRGCN 将其用于检测异常系统行为图。与
TraceLog 一样，本项目使用作者发布的预处理图文件，没有从原始网络流或系统调用
记录重新生成该数据集。

### 6.2 已发布图文件

文件结构与 TraceLog 基本相同，但有两处关键差异：

1. 节点特征维度为 26；
2. 发布的 `edge_index.csv` 没有 `edge_type` 列。

加载器在缺失 `edge_type` 时将所有边设为类型 0，因此当前模型输入是：

```text
8 种节点类型 + 1 种显式边类型 + 26 维节点特征
```

26 维特征列不能被重新解释为 26 种显式边类型。

### 6.3 标签与划分

| 划分 | 图数 | 正常 | 异常 |
|---|---:|---:|---:|
| 训练 | 375 | 375 | 0 |
| 独立验证 | 0 | 0 | 0 |
| 测试/评估 | 225 | 125 | 100 |

发布文件只有训练与评估两份图 ID。HRGCN 论文描述了训练、验证、测试划分，但
当前公开文件没有独立验证文件。论文实验设置必须按实际文件说明这一差异。

### 6.4 数据捷径风险

只使用节点数、边数和类型计数，One-Class SVM 已达到约
`0.9984 AUROC / 0.9981 AP`。因此 FlowGraph 上接近 1.0 的结果可能主要来自
图规模或分布差异，不能单独证明模型学到了关系偏差注意力。

## 7. HDFS-100k 派生图数据集

### 7.1 原始数据来源

当前使用 Zenodo 发布的：

- `HDFS_100k.log_structured.csv`；
- HDFS_v1 block 级 `anomaly_label.csv`。

结构化日志字段包括 `Content`、`EventId` 和 `Component`。原始异常标签由
HDFS 数据发布方提供，本项目没有人工重新标注。

该数据只包含 10 万条日志，用于判断构图和模型链路是否可用。最终论文若与完整
HDFS 文献结果比较，必须重新处理 Loghub 完整 HDFS_v1，并将名称写为
`HDFS_v1`；当前结果必须写为 `HDFS-100k`。

### 7.2 样本单元

HDFS 日志消息中包含形如：

```text
blk_-1608999687919862906
```

的 block ID。同一 block ID 对应的日志事件按原 CSV 顺序组成一条事件序列，并
转换为一张图。因此：

```text
一张图 = 一个 HDFS block 的事件轨迹
一个节点 = 一次日志事件出现
图标签 = 该 block 的官方 Normal/Anomaly 标签
```

例如派生图 0 对应：

```text
source_id      blk_-1001553972418305662
label          0
sequence_len   13
```

### 7.3 词表构建

1. 读取结构化日志；
2. 按字符串排序全部已观察 `EventId`，映射为连续整数；
3. 按字符串排序全部已观察 `Component`，映射为辅助整数；
4. 每个事件节点的 `node_type` 等于其 `EventId` 编号。

HDFS-100k 中共观察到 19 种事件类型。

### 7.4 节点特征

设一个 block 序列长度为 `L`，第 `i` 个节点的事件类型为 `t_i`，组件编号为
`c_i`。8 维特征定义为：

| 维度 | 定义 | 含义 |
|---|---|---|
| 1 | `i / max(L-1,1)` | 归一化时间位置 |
| 2 | `log(1+L) / 16` | 序列长度尺度 |
| 3 | `seen_i(t_i) / count(t_i)` | 当前类型出现进度 |
| 4 | `count(t_i) / L` | 当前类型在序列中的频率 |
| 5 | `1[i=0]` | 首节点标志 |
| 6 | `1[i=L-1]` | 尾节点标志 |
| 7 | `(c_i mod 997) / 997` | 组件编号的缩放表示 |
| 8 | `log(1+t_i) / 16` | 事件类型编号的缩放表示 |

其中 `seen_i(t_i)` 是到位置 `i` 为止类型 `t_i` 已出现的次数。

### 7.5 边构建

对每条事件序列构造三种有向边：

| 边类型 | 构建规则 | 表达含义 |
|---|---|---|
| 0 | `i-1 -> i` | 相邻事件的时间顺序 |
| 1 | `previous_same_event -> i` | 同一 EventId 的重复依赖 |
| 2 | `previous_same_component -> i` | 同一组件的连续行为 |

若一张图只有一个节点，则增加类型 0 的自环，避免空边图。

HDFS 使用 canonical 关系三元组，最大关系数为：

```text
19 * 3 * 19 = 1083
```

### 7.6 标签和划分

官方 `Normal` 转换为 0，其他标签转换为 1。seed 42 下：

| 划分 | 图数 | 正常 | 异常 |
|---|---:|---:|---:|
| 训练 | 4576 | 4576 | 0 |
| 验证 | 1525 | 1525 | 0 |
| 测试 | 1839 | 1526 | 313 |

正常图按 60%/20%/20% 随机划分，所有异常图只进入测试集。

## 8. ADFA-LD 派生图数据集

### 8.1 原始数据来源

ADFA-LD 是面向 Linux 主机入侵检测的系统调用轨迹数据集，包含：

- `Training_Data_Master`：833 条正常训练轨迹；
- `Validation_Data_Master`：4372 条正常验证轨迹；
- `Attack_Data_Master`：746 条攻击轨迹。

攻击轨迹分为六类：

| 攻击类别 | 轨迹数 |
|---|---:|
| Hydra-SSH | 176 |
| Hydra-FTP | 162 |
| Java Meterpreter | 124 |
| Web Shell | 118 |
| Adduser | 91 |
| Meterpreter | 75 |

原始轨迹已经用整数表示系统调用，本项目不修改攻击类别和标签。

### 8.2 样本单元

每个文本文件是一条系统调用序列，并被转换为一张图：

```text
一张图 = 一个系统调用轨迹文件
一个节点 = 一次系统调用出现
图标签 = 正常目录为 0，Attack_Data_Master 为 1
```

例如派生图 0：

```text
source_file    Training_Data_Master/UTD-0001.txt
label          0
sequence_len   819
```

派生图 5205：

```text
source_file    Attack_Data_Master/Adduser_1/UAD-Adduser-1-1371.txt
label          1
sequence_len   279
```

### 8.3 训练词表与未知类型

系统调用词表只使用 `Training_Data_Master` 中的正常训练轨迹建立：

1. 收集正常训练轨迹中的 syscall ID；
2. 排序并映射到 `1...K`；
3. 编号 0 保留为 unknown；
4. 验证和攻击轨迹中未见过的 syscall 映射为 0。

这一过程不读取攻击标签来创建输入词表，避免用测试攻击轨迹扩充类别空间。最终
图数据包含 151 种节点类型（含 unknown）。

### 8.4 节点特征

ADFA-LD 使用与 HDFS 相同的 8 维位置和频率特征。由于没有组件字段，第 7 维
辅助编号特征当前恒为 0；其余维度中的 `t_i` 表示训练词表映射后的 syscall 类型。

### 8.5 边构建

当前派生数据文件包含：

| 边类型 | 当前规则 |
|---|---|
| 0 | 相邻 syscall：`i-1 -> i` |
| 1 | 上一次相同 syscall 到当前 syscall |
| 2 | 上一次相同辅助编号到当前节点 |

由于 ADFA-LD 没有辅助编号，当前所有辅助编号均为 0，因此类型 2 与相邻顺序边
基本重复。**这一关系不应被描述为新的业务语义。最终论文实验建议移除类型 2，
重新生成两类边的 ADFA-LD 图，并重新运行实验。**

### 8.6 高基数节点类型适配

若使用 canonical 关系：

```text
151 * 3 * 151 = 68403
```

种关系会导致关系矩阵参数在模型创建阶段占用十余 GB 显存。因此当前 ADFA-LD
配置采用：

```yaml
model:
  relation_schema: edge_only
```

系统调用类型仍通过 151 个类型专属输入投影区分，消息变换和原型只按边类型共享。
这不是把所有 syscall 合并为一种节点，而是避免为每一对 syscall 额外学习一张
`H x H` 关系矩阵。

### 8.7 标签和划分

| 划分 | 图数 | 正常 | 异常 |
|---|---:|---:|---:|
| 训练 | 833 | 833 | 0 |
| 验证 | 2186 | 2186 | 0 |
| 测试 | 2932 | 2186 | 746 |

训练目录全部作为正常训练集。4372 条正常 validation 轨迹在 seed 42 下等分为
验证正常样本和测试正常样本；746 条攻击轨迹全部进入测试集。

## 9. Packed 派生图文件的制作

HDFS 和 ADFA-LD 在构图后写成可内存映射的 packed 格式：

| 文件 | 内容 |
|---|---|
| `x.npy` | 所有图节点特征连续拼接 |
| `node_type.npy` | 所有节点类型连续拼接 |
| `edge_index.npy` | 所有边按全局节点偏移存储 |
| `edge_type.npy` | 所有边类型连续拼接 |
| `graph_ptr.npy` | 每张图的节点起止偏移 |
| `edge_ptr.npy` | 每张图的边起止偏移 |
| `labels.npy` | 图标签 |
| `graph_ids.npy` | 图 ID |
| `train_ids.txt` | 训练图 ID |
| `validation_ids.txt` | 验证图 ID |
| `test_ids.txt` | 测试图 ID |
| `metadata.json` | 图数、类型数、特征维数等统计 |
| `samples.csv` | 派生图与原始 block/文件的追踪映射 |

第 `g` 张图的节点切片为：

```text
graph_ptr[g] : graph_ptr[g+1]
```

边切片为：

```text
edge_ptr[g] : edge_ptr[g+1]
```

读取时再减去该图的节点全局偏移，恢复局部 `edge_index`。这种格式避免将数千张
图同时反序列化为 Python 对象。

## 11. 可直接改写进论文的表述

### 11.1 数据集总述

> 为评估模型在不同系统行为图上的泛化能力，本文采用 TraceLog、FlowGraph、
> HDFS 和 ADFA-LD 四个数据集。TraceLog 和 FlowGraph 使用 HRGCN 作者发布的
> 预处理图数据；对于原始形式为结构化日志和系统调用序列的 HDFS 与 ADFA-LD，
> 本文进一步构建事件出现级有向图。所有数据统一为单类图级异常检测设置，训练
> 阶段只使用正常图，异常标签仅用于测试阶段的性能评估。

### 11.2 HDFS 构图表述

> 对于 HDFS，每个 block ID 对应一张图，每条结构化日志事件对应一个节点。
> 节点类型由日志模板 EventId 决定，节点属性由事件位置、序列长度、模板出现
> 频率、组件编号等统计量组成。本文分别建立相邻事件边、同模板历史依赖边和
> 同组件历史依赖边，以同时保留日志序列的时序结构与重复行为模式。图标签沿用
> HDFS 数据集提供的 block 级异常标签。

当前只能在该段后注明“使用 HDFS-100k 子集”。处理完整 HDFS_v1 后才能删除
“子集”限制。

### 11.3 ADFA-LD 构图表述

> 对于 ADFA-LD，每条系统调用轨迹对应一张图，每次系统调用出现对应一个节点。
> 系统调用词表仅由正常训练轨迹构建，验证和测试阶段未见过的调用映射为 unknown。
> 节点属性编码系统调用在轨迹中的相对位置、出现进度和频率等信息；有向边用于
> 描述相邻调用关系和相同调用的历史依赖。训练目录中的正常轨迹用于模型训练，
> 攻击目录中的六类攻击轨迹仅用于测试。

这段采用建议的两类边表述。只有在移除当前冗余类型 2 并重跑实验后，才能原样
写入最终论文。

## 12. 当前审计限制

### 12.1 HDFS-100k

- 当前不是完整 HDFS_v1，不能与完整数据上的 SOTA 数字直接比较；
- 当前实现使用正则 `str.extract`，一行包含多个 block ID 时只保留第一个；
- EventId 和 Component 字典当前由整个 100k 子集建立，属于无标签转导式预处理；
- 最终严格归纳实验建议先确定正常训练 block，再只用训练日志建立词表。

### 12.2 ADFA-LD

- 当前类型 2 边因辅助编号恒为 0，与顺序边基本重复；
- 最终论文应移除该边、更新 `num_edge_types=2` 并重跑全部 ADFA-LD 实验；
- `edge_only` 是高基数类型适配，不等同于 TraceLog 使用的 canonical 关系。

### 12.3 TraceLog 和 FlowGraph

- 两者缺少可审计的原始日志到图生成代码，本项目无法证明完全重现上游构图；
- TraceLog 的 7 个匿名属性不能被自行赋予业务名称；
- FlowGraph 发布文件与论文关于 26 种边类型的描述不一致；
- FlowGraph 存在很强的图规模/类型计数捷径。
