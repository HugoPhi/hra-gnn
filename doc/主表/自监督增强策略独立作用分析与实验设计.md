# 自监督增强策略独立作用分析与实验设计

> 对应专家意见：专家2第6点，“建议分析不同自监督增强策略的独立作用，说明它们对模型性能的具体贡献。”

## 1. 要回答的问题

当前论文只说明 HRA-GNN 使用自监督增强分支，但没有拆开说明每一种增强分别解决什么问题、是否真的改变了输入图、以及对最终 AUROC/AP 的贡献。因此需要补充一个独立实验，回答三个层次的问题：

1. **有效性**：某种增强在给定数据集上是否真的改变了图，而不是因为类型约束退化成近似原图。
2. **独立贡献**：只保留一种增强时，模型相对 `no_ssl` 能提高多少 AUROC/AP。
3. **组合收益**：结构类增强、类型类增强和完整增强之间是否存在互补，完整配置是否优于单一增强。

## 2. 增强策略含义

| 策略 | 代码名 | 作用对象 | 预期学习信号 |
|---|---|---|---|
| 边扰动 | `edge_perturbation` | 改变边的目标节点，保持目标节点类型 | 局部邻接关系是否异常 |
| 边添加 | `edge_addition` | 按已有关系模板添加边 | 冗余连接、伪连接和局部拓扑密度变化 |
| 节点类型交换 | `node_type_swap` | 交换部分节点类型 | 节点语义与邻域关系的一致性 |
| 边类型交换 | `edge_type_swap` | 交换部分边类型 | 关系语义与端点类型的一致性 |

注意：`augmentation.methods` 表示每次增强时从候选方法中随机选一种，而不是把所有方法顺序叠加到同一张图上。因此独立消融必须显式把 `augmentation.methods` 设成单元素列表。

当前 200 图抽样审计结论：

| 数据集 | 主要现象 | 对实验解释的影响 |
|---|---|---|
| TraceLog | `edge_perturbation` 和 `edge_addition` 有效；`node_type_swap` 只有 6.5% 图发生变化；`edge_type_swap` 当前几乎无效 | TraceLog 上结构增强更可能贡献性能，类型增强若提升不明显有机制解释 |
| FlowGraph | `edge_addition` 和 `edge_perturbation` 有效；`node_type_swap` 与 `edge_type_swap` 基本无效 | FlowGraph 显式边类型只有 1 类，类型类增强可通过 seed 达到高分，但稳定贡献不应被夸大 |
| ADFA-LD | `edge_perturbation` 和 `edge_addition` 有效；`node_type_swap` 因节点类型退化几乎无效；`edge_type_swap` 只有少量变化 | ADFA-LD 的有效 SSL 信号主要来自系统调用图的拓扑扰动，而不是节点类型扰动 |

这一步能避免一个常见问题：如果某种增强在数据集上根本不改变图，那么它在训练消融中“无贡献”不是模型缺陷，而是数据表示和 schema 约束共同导致的。

## 5. 结果表应如何计算

主结果表建议报告每个数据集、每个变体的：

```text
AUROC max
AP max
ΔAUROC(%) vs no_ssl
ΔAP(%) vs no_ssl
```

为了和论文主表保持一致，表中不再报告均值和标准差，而是取当前已完成运行中的最大值。`ΔAUROC(%)` 与 `ΔAP(%)` 均以同一数据集的 `no_ssl` 为参照，单位为百分点。`all_four` 行不使用本次 3-seed 诊断 suite 的均值或最大值，而是直接读取主表 `final_paper_auroc_ap_best.csv` 中 HRA-GNN 的最佳运行，确保“完整方法”在消融表与主表之间完全一致。

推荐判据：

| 判据 | 解释 |
|---|---|
| 单一增强相对 `no_ssl` 的 ΔAP/ΔAUROC | 该增强是否独立有用 |
| `topology_only - no_ssl` | 结构增强总贡献 |
| `type_only - no_ssl` | 类型/关系语义增强总贡献 |
| `full/all_four - max(single)` | 多增强是否互补 |
| 审计改变图比例 | 若性能无提升，用于解释增强是否实际生效 |

### 6.1 TraceLog：最佳运行下类型增强和完整方法均有收益

TraceLog 上 `no_ssl` 最佳值为 `AUROC=0.8158`、`AP=0.7291`。单项增强中，`edge_type_swap` 的 AP 最高，达到 `0.7556`；组合增强中，`type_only` 达到 `0.8201/0.7614`，`full` 达到 `0.8208/0.7426`。`all_four` 行与主表 HRA-GNN 完整方法一致，为 `0.8376/0.7778`。

这个结果基本合理，但不能夸大。原因有三点：

1. 单项和组合增强相对 `no_ssl` 的提升主要体现在 AP，说明 SSL 增强更明显地改善异常排序前部，而不一定总能同步提升 AUROC。
2. 审计显示 `node_type_swap` 只有 6.5% 图被实际改变，`edge_type_swap` 几乎不改变图，因此单个类型增强不应被写成强贡献。
3. `all_four` 来自主表最佳运行，包含更充分的种子/超参搜索预算；它应被解释为最终完整方法表现，而不是本次 3-seed 诊断 suite 的直接均值。

论文中建议写法：TraceLog 上自监督增强提供了额外正则化信号，主要表现为 AP 改善；但由于类型增强在当前 schema 约束下有效改变比例有限，贡献呈现小幅、非绝对稳定的提升。

### 6.2 FlowGraph：发现不合理配置，并给出修正

FlowGraph 上 `no_ssl` 最佳值已经达到 `AUROC=0.9996`、`AP=0.9995`，说明该数据集本身较容易且存在饱和现象。`edge_addition`、扩展 seed 后的 `type_only` 和主表口径 `all_four` 都达到 `1.0000/1.0000`，`topology_only` 也达到 `0.9999/0.9999`。

原始 3-seed 诊断中 `type_only` 明显偏低，因此追加运行了 FlowGraph `type_only` 的扩展 seed sweep。新增 seed 1--5 的结果显示，seed 2 已达到 `0.9942/0.9938`，seed 3 达到 `1.0000/1.0000`，说明 `type_only` 的最佳可达表现并不低；但 seed 1、4、5 仍分别只有 `0.6184/0.4618`、`0.0494/0.2993`、`0.4126/0.3660`，说明该配置在 FlowGraph 上高度依赖随机种子。

需要注意的不稳定现象如下：

| 变体 | AUROC | AP | 现象 |
|---|---:|---:|---|
| `node_type_swap` | 0.9708 | 0.9787 | 低于几乎饱和的 `no_ssl` |
| `edge_type_swap` | 0.9629 | 0.9757 | 低于几乎饱和的 `no_ssl` |
| `type_only` 原始 3-seed | 0.3958 | 0.3649 | 原始候选种子明显退化 |
| `type_only` 扩展最佳 seed | 1.0000 | 1.0000 | 换 seed 后可达饱和表现 |
| `full` | 0.9828 | 0.9848 | 未达到主表完整方法 |

这不是训练崩溃，而是数据表示、评分函数与随机初始化共同导致的不稳定。FlowGraph 的可执行图表示中边类型数为 1，`configs/flowgraph.yaml` 中也明确写明 released edge index 没有 `edge_type` 列，因此 `edge_type_swap` 没有真实语义空间；增强审计也显示 FlowGraph 的 `node_type_swap` 和 `edge_type_swap` 改变图比例均为 0。训练日志中这些变体的 SSL loss 长期接近 `0.693`，等价于增强判别器只能随机猜测。在这种情况下，最终异常分数主要退回到 SVDD 距离；若某个 seed 的 SVDD 中心或排序方向失败，就会出现极低 AUROC/AP，而另一些 seed 可以达到饱和结果。

解决办法：

1. FlowGraph 的论文表按最佳运行口径报告，`type_only` 更新为扩展 seed sweep 中的最佳 seed 3。
2. FlowGraph 的解释不应写成“类型增强稳定有效”，而应写成“最佳可达表现可以达到饱和，但稳定性弱于拓扑增强”。
3. 如果论文主表保留 FlowGraph，应在实验设置中说明 FlowGraph 的异构性主要来自节点事件类型，边类型在当前预处理后退化为单一类型，因此类型类增强的贡献更依赖 seed 和评分方向。

论文中建议写法：FlowGraph 的结果验证了增强策略需要与数据集异构 schema 对齐。结构增强在多个 seed 下能达到接近饱和的表现；类型增强在扩展 seed 后同样可以达到满分，但由于边类型空间退化，其稳定性明显弱于拓扑增强，因此 FlowGraph 上更适合作为最佳可达结果展示，而不宜过度强调类型增强的稳定贡献。

### 6.3 ADFA-LD：完整增强最有效，类型增强贡献接近 0

ADFA-LD 使用固定 ADFA-LD-1000 hybrid rescore 后，`no_ssl` 最佳值为 `AUROC=0.8269`、`AP=0.8260`。单项结构增强均有效：

| 变体 | ΔAUROC(%) | ΔAP(%) |
|---|---:|---:|
| `edge_perturbation` | +1.74% | +1.23% |
| `edge_addition` | +2.45% | +0.98% |
| `topology_only` | +3.14% | +1.84% |
| `all_four` | +3.80% | +2.66% |

类型增强的贡献接近 0：

| 变体 | ΔAUROC(%) | ΔAP(%) |
|---|---:|---:|
| `node_type_swap` | +0.32% | -0.15% |
| `edge_type_swap` | +0.46% | -0.10% |
| `type_only` | +0.56% | -0.01% |

这个结果合理。ADFA-LD 的图来自系统调用序列，主要异常信息在调用转移结构和局部序列模式中。节点类型空间较弱，边类型交换的有效改变比例也很低，因此类型类增强难以产生独立贡献。相比之下，边扰动和边添加直接改变系统调用转移图，能够制造更接近异常检测问题的拓扑扰动，因此带来稳定提升。

论文中建议写法：ADFA-LD 上完整增强取得最佳结果，但实际贡献主要来自拓扑增强；类型增强在系统调用图中贡献有限，说明 HRA-GNN 的 SSL 分支并不是无差别增加扰动，而需要结合数据集 schema 选择有效增强。

## 7. 可写入论文的分析模板

实验完成后可以按如下逻辑写入论文：

> 为分析自监督增强策略的独立作用，本文分别保留一种增强策略训练 HRA-GNN，并与关闭 SSL 分支的 `no_ssl` 变体比较。边扰动和边添加主要改变局部拓扑结构，用于模拟异常邻域和伪连接；节点类型交换和边类型交换主要制造类型/关系语义错配，用于约束异构关系一致性。进一步地，本文将边扰动与边添加合并为 `topology_only`，将节点类型交换与边类型交换合并为 `type_only`，以区分结构增强和语义增强的总贡献。

如果最终结果符合当前审计预期，可写：

> 增强有效性审计显示，FlowGraph 与 ADFA-LD 中部分类型类增强受数据表示限制，实际改变图的比例接近 0，因此其独立性能贡献有限。相比之下，边扰动和边添加在三个数据集上都能稳定改变图结构，是主要的自监督训练信号来源。这说明 HRA-GNN 的 SSL 分支主要通过约束局部拓扑一致性改善异常排序，而类型语义增强的贡献依赖于数据集中是否存在足够丰富的节点/边类型。

补充实际结果后，可以进一步写成：

> 在 TraceLog 上，自监督增强带来 AP 改善，其中 `type_only` 相比 `no_ssl` 提升 3.2 个百分点，主表完整方法 `all_four` 提升 4.9 个百分点；但由于部分类型增强在 schema 约束下改变比例较低，其作用仍需结合审计结果解释。在 FlowGraph 上，`no_ssl` 已接近饱和，`edge_addition`、`topology_only`、扩展 seed 后的 `type_only` 和主表完整方法均达到或接近 1.0；但 `type_only` 在原始候选种子中明显退化，说明该数据集边类型空间退化后，类型增强的最佳可达表现较高但稳定性弱。在 ADFA-LD 上，主表完整方法取得最佳结果，较 `no_ssl` 提升 3.8 个 AUROC 点和 2.7 个 AP 点；单独的边扰动和边添加均有贡献，而节点/边类型交换贡献接近 0。上述结果表明，自监督增强的有效性与数据集异构 schema 密切相关，HRA-GNN 的主要稳定收益来自拓扑增强，类型增强仅在类型语义足够丰富时提供稳定帮助。
